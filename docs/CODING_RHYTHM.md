# 代码审计与修复规范 — 四步闭环

> **审计 → 诊断 → 执行 → 验证**
>
> 这是 2026-05-26 迭代中验证有效的工作模式。
> 其核心是：不用感觉判断，用工具扫描；不零散修，按五类归类；修完不是结束，清零才算完。

## 一、审计（Audit）：用代码扫描代替人肉猜测

不说"我觉得这里可能有问题"，而是用工具扫描出确凿的证据。

| 工具 | 用途 | 示例 |
|------|------|------|
| `grep -rn` | 全量扫描关键词、函数调用 | 扫描 `session` 发现 3 个文件各自定义变量 |
| `go test ./...` | 确认当前测试基线 | 21 包全部通过后再动手 |
| 走读 >100 行函数 | 理解控制流 | OnMessage 178 行大状态机 |
| 检查调用链 | 发现断路 | `GenerateSessionSummary` 从不自动触发 |

**审计清单：**
- [ ] `grep -rn` 扫描核心关键词，绘制调用关系
- [ ] 走读 >100 行的函数，标记每个分支的触发条件
- [ ] 列出所有 `TODO` 注释，区分"待实现债务"和"已过时残留"
- [ ] `go test ./...` 记录当前基线

## 二、诊断（Diagnose）：按五类模式归类

不零散列问题，用五个标准模式覆盖。

| 归类 | 判断标准 | 示例 |
|------|---------|------|
| **割裂** | 同一概念在多个位置独立定义 | 3 个文件各自定义 session 关键词 |
| **混杂** | 一个函数承担多个不相关职责 | OnMessage 中系统指令和对话混在一起 |
| **断路** | 功能已实现但未接入主流程 | `GenerateSessionSummary` 从不到自动调用 |
| **不一致** | 同类操作执行标准不同 | `review_handler` 绕过硬化层 vs 其他插件走 `ValidateAndExecute` |
| **死代码** | 完整实现但零调用 | `internal/legacy/aliases.go` |

**诊断格式：** 每项记录四要素：`位置 | 现象 | 影响范围 | 修复方向`。2-3 行即可。

## 三、执行（Fix）：按依赖顺序修复

修复顺序不是随机的。先消除割裂（建立中心），再拆分混杂（清理路径），然后接通断路（激活能力），最后规范不一致和清理死代码。

| 顺序 | 动作 | 示例 |
|------|------|------|
| **1. 统一** | 散落的知识集中到一个文件 | `intent_keywords.go` 统一 3 个位置的词表 |
| **2. 拆分** | 大函数按职责拆小 | `OnMessage` 178→26 行，纯分发器 |
| **3. 接通** | 已实现的功能自动触发 | `GenerateSessionSummary` 异步自动调 |
| **4. 规范** | 统一硬化层路径 | `review_handler` 改走 `ValidateAndExecute` |
| **5. 清理** | 删除死代码 | `internal/legacy/` 整包删除 |

**执行原则：**
- 每次只修一类，修完验证再下一类
- 建立中心词表/注册表是消除割裂的万能模式
- 拆分大函数到 30 行以下，每个函数只做一件事

## 四、验证（Verify）：三种方式交叉确认

不用"我觉得修好了"判定，用工具交叉验证。

| 验证方式 | 命令 | 示例 |
|----------|------|------|
| 编译检查 | `go build ./...` | ✅ |
| 测试回归 | `go test ./...` | 22 包全部通过 |
| 反向扫描 | `grep -rn "旧模式"` | 残留的旧关键词/旧路径/旧调用 |

**验证清单：**
- [ ] `go build ./...` ✅
- [ ] `go test ./...` ✅（全部包）
- [ ] 修复割裂 → 扫描旧关键词是否还有残留定义
- [ ] 修复混杂 → 大函数行数降到 30 以下
- [ ] 修复断路 → 功能在正确时机自动触发
- [ ] 修复不一致 → 扫描是否还有直接文件系统调用
- [ ] 修复死代码 → 已删包无调用方残留
- [ ] 记录 `DEVLOG_`，四要素：位置、现象、影响、修复方向

```markdown
## 审计扫描

grep 扫描结果：
- 关键词 X 在文件 A/B/C 中有定义（割裂）
- 关键词 Y 有定义但无调用（死代码）

>100 行函数：
- funcX(): 180 行，包含 A/B/C 三个职责（混杂）

TODO 清单：
- // TODO: implement X — 2026-01 遗留（过时残留）
- // FIXME: 硬化层绕过 — 2026-05（待修复的债务）

## 诊断归类

| 归类 | 位置 | 现象 | 影响 | 修复 |
|------|------|------|------|------|
| 割裂 | A/B/C | X 定义 3 份 | 改一漏二 | 建立中心 |
| 混杂 | funcX | 180 行 3 职责 | 难以测试 | 拆成 3 个 |
```
