## 任务 A：遗留决策关闭 — 343 条 .sy 单块结构是否迁移多块

### 背景

05-28 吸收时记录了一个待决策项：343 条 `.sy` 文件在 Lute 集成前迁移，全部为单块（`type: paragraph`，内含原始 Markdown）。是否用 Lute 重跑迁移为多块结构？

### 分析

逐项核查搜索评分代码（`knowledge.go:391-397`）：

```go
for _, block := range entry.BlockContents {
    if strings.Contains(blockLower, q) || stringContainsAny(blockLower, q) {
        score += 2
        break  // ← 命中即退出
    }
}
```

`break` 决定了无论有 1 个块还是 10 个块，BlockContents 最多贡献 +2 分。多块结构对主搜索管道零收益。L1 embedding 存储在 `doc.Embedding`，与块数无关。检索基线 8/10 已达阶段一目标。

### 决策：**永久顺延，不迁移**

触发重新评估的条件：评分代码升级为 heading 块 +3、paragraph 块 +2 的差异权重。决策已存入知识库（`claude_dev` namespace）。

## 任务 B：SiYuan 吸收产物全量审计

审计范围：`storage.go` / `link_index.go` / `knowledge_graph.go` / `markdown_blocks.go` + 集成点 `knowledge.go`。

发现 **3 个 P0 Bug + 2 个 P1 性能问题**，全部修复。

### P0-1：`findKnowledgeByRawRefLocked` 扫 JSON 目录，BlockStorage 下去重永远失效

**文件**：`knowledge.go:285-301`

**问题**：函数扫 `knowledgeDir/*.json`（旧格式目录），但 BlockStorage 下所有新条目在 `notebooks/*.sy`。结果：`knowledge_remember` / `knowledge_add` 的去重检查永远返回 nil，相同 `raw_ref` 可反复写入，产生重复条目。

**修复**：改用 `Storage().AllEntries()` 全量扫描，与当前存储后端无关。

```go
// 修复前
entries, _ := os.ReadDir(knowledgeDir)
for _, e := range entries {
    entry := loadKnowledge(strings.TrimSuffix(e.Name(), ".json"))
    ...
}

// 修复后
for _, entry := range Storage().AllEntries() {
    if entry.RawRef == rawRef {
        return entry
    }
}
```

### P0-2：`SaveEntry` 更新路径丢 Tags / Namespace / SourceType

**文件**：`storage.go:370-378`

**问题**：更新已有文档时只更新 `Title`/`UpdatedAt`/`Embedding`，`Tags`/`Namespace`/`SourceType` 全部静默丢弃。`knowledge_update` 修改标签后下次加载回来是旧标签。

**修复**：更新路径补齐全部结构字段，保留 `Blocks`/`Refs`/`Backlinks` 不变（避免用重建的截断 summary 覆盖原始内容）。

注：未重新生成 Blocks 是有意为之——`DocToEntry` 重建 summary 时截断至 300 rune，若每次 update 都重建，summary 会渐进丢失。

### P0-3：`DocToEntry` 按字节截断 summary，截断 UTF-8 汉字

**文件**：`storage.go:244`

**问题**：`summary[:300]` 按字节切，中文 3 字节/字符，截断位置可能落在字符中间 → 输出乱码。

**修复**：改用 `[]rune` 转换后截断。

```go
// 修复前
if len(summary) > 300 {
    summary = summary[:300]
}

// 修复后
if runes := []rune(summary); len(runes) > 300 {
    summary = string(runes[:300])
}
```

### P1-1：`SaveEntry` 每次写都重新读全部 345 个 .sy 文件

**文件**：`storage.go:387-405`

**问题**：backlinks 更新逻辑无条件执行 `loadDocIndex(s.dir)`（读 345 文件），再把有链接的文件全部回写，最后重建内存索引。当前知识库里没有 `[[wikilink]]` 语法，O(N) IO 每次白跑。以 `knowledge_remember` 为例，一次写入触发 ~345 次磁盘读。

**修复**：先扫块内容是否含 `[[` 或 `((`，无链接时跳过全量 IO。

```go
hasLinks := false
for _, b := range doc.Blocks {
    if strings.Contains(b.Content, "[[") || strings.Contains(b.Markdown, "[[") ||
        strings.Contains(b.Content, "((") || strings.Contains(b.Markdown, "((") {
        hasLinks = true
        break
    }
}
if hasLinks {
    // 原有 loadDocIndex + UpdateBacklinks 逻辑
}
```

### P1-2：`BuildLocalGraph/BuildGlobalGraph` 绕开内存索引，每次直接读磁盘

**文件**：`knowledge_graph.go:29, 93`

**问题**：两个图谱工具直接调 `loadDocIndex(nd)`，完全忽视 `BlockStorage.index` 已有的 345 条内存缓存。每次图谱查询触发 345 次磁盘读。

**修复**：新增 `BlockStorage.DocMap()` / `blockDocMapForDir(nd)` 方法，图谱函数优先用内存快照，仅在 dir 不匹配（测试场景）时降级到 `loadDocIndex`。

**实现陷阱**：最初用 `Storage()` 触发存储初始化，导致图谱测试的临时 BlockStorage 污染全局 `currentStorage`，`TestRetrievalQualitySuite` 从测试临时目录（3 条 docs）读数据。修复：`blockDocMapForDir` 改用 `storageMu.Lock()` + 直接读 `currentStorage`，不触发初始化。
