## 任务 A：遗留决策关闭 — 343 条 .sy 单块结构是否迁移多块

### 背景

05-28 吸收时记录了一个待决策项：343 条 `.sy` 文件在 Lute 集成前迁移，全部为单块（`type: paragraph`，内含原始 Markdown）。是否用 Lute 重跑迁移为多块结构？

### 分析

逐项核查搜索评分代码（`knowledge.go:391-397`）：

```go
for _, block := range entry.BlockContents {
    if strings.Contains(blockLower, q) || stringContainsAny(blockLower, q) {
        score += 2
        break  // ← 命中即退出
    }
}
```

`break` 决定了无论有 1 个块还是 10 个块，BlockContents 最多贡献 +2 分。多块结构对主搜索管道零收益。L1 embedding 存储在 `doc.Embedding`，与块数无关。检索基线 8/10 已达阶段一目标。

### 决策：**永久顺延，不迁移**

触发重新评估的条件：评分代码升级为 heading 块 +3、paragraph 块 +2 的差异权重。决策已存入知识库（`claude_dev` namespace）。

### P1-1：`SaveEntry` 每次写都重新读全部 345 个 .sy 文件

**文件**：`storage.go:387-405`

**问题**：backlinks 更新逻辑无条件执行 `loadDocIndex(s.dir)`（读 345 文件），再把有链接的文件全部回写，最后重建内存索引。当前知识库里没有 `[[wikilink]]` 语法，O(N) IO 每次白跑。以 `knowledge_remember` 为例，一次写入触发 ~345 次磁盘读。

**修复**：先扫块内容是否含 `[[` 或 `((`，无链接时跳过全量 IO。

```go
    if strings.Contains(b.Content, "[[") || strings.Contains(b.Markdown, "[[") ||
        strings.Contains(b.Content, "((") || strings.Contains(b.Markdown, "((") {
        hasLinks = true
        break
    }
}
if hasLinks {
    // 原有 loadDocIndex + UpdateBacklinks 逻辑
}
```

## 任务 C：可靠性 + 简化三连（R1 / R2 / S1 + 死代码清理）

### 动机

确立一条非功能性的优化主线：**可靠性 → 可用性 → 简单便用性**，明确不加功能。
顺序定为 R1（panic 安全）→ R2（测试隔离）→ S1（拆上帝文件）。
逻辑是：先保证进程不会被单点 panic 掀翻（可靠性底线），再保证测试不互相污染
（否则后续任何重构都无法用测试背书），最后才动结构（拆文件）。次序不可换——
没有可信的测试隔离就去拆 3578 行文件，等于盲拆。

### R1：panic 兜底（commit `1592f26`，`internal/observatory/recover.go`）

**问题**：goroutine 里的 panic 不会被调用方的 recover 捕获，会直接终止整个进程。
agent 并行子任务、workflow 执行器、HTTP handler 里任何一个未预期 panic 都能掀翻
线上服务。此前没有统一的兜底机制。

**方案**：在 observatory 层（不是 kernel——kernel 冻结）提供三件套：

| API | 用途 |
|-----|------|
| `Recover(context)` | deferred 陷阱，记栈 + 发 `EventPanicRecovered` 事件，不 re-panic |
| `RecoverWith(context, cleanup)` | Recover + 仅在 panic 时跑的清理回调（用于回填失败结果） |
| `SafeGo(context, fn)` | 启动一个被 Recover 守护的 goroutine（fire-and-forget） |

panic 被捕获后统一走 `reportPanic`：抓 64KB 栈、`log.Printf` 带 `[panic-recovered]`
前缀、`PublishEvent` 发 `system.panic_recovered`，订阅方可告警。

**关键陷阱（务必记住）— LIFO defer 次序**：
持有"完成契约"的 goroutine（WaitGroup.Done / 往 result channel 发值）必须把
Done/send 的 defer **注册在 Recover 之前**。Go 的 defer 是 LIFO，后注册的先跑——
所以 Recover 先执行吞掉 panic，随后 Done/send 仍会执行，契约不被破坏。
若顺序写反，panic 时 Done 永不执行 → WaitGroup 永久阻塞 → 死锁。
`RecoverWith` 就是为这种场景设计的：在 cleanup 里用 recovered 值回填失败结果。

**接入点（非测试调用点，全部已验证）**：
- `cmd/beishan/main.go`（+30）— HTTP handler 兜底
- `internal/agent/runner.go`（+11）— 并行子 agent
- `internal/workflow/engine.go`（+7）/ `gods_executor.go`（+2）— 两套工作流执行器
- 测试 `recover_test.go`（62 行）覆盖三件套 + LIFO 契约不破坏

**为什么放 observatory 而非 kernel**：kernel 冻结；且 observatory 本就持有 event bus
（`PublishEvent`），panic 事件天然属于"可观测性"范畴。

### R2：测试隔离（commit `1592f26`，`internal/tools/storage_test.go`）

**问题（根因，不是表象）**：`Storage()` 用 `currentStorage` 做永久懒缓存，**没有 reset 路径**。
测试里只还原 `knowledgeDir` 是不够的——第一个调用 `Storage()` 的测试会把某个目录的
存储缓存进 `currentStorage`，后续测试即便改了 `knowledgeDir` 也读不到新目录，而是
命中旧缓存。这正是任务 B 里 P1-2 审计时踩的"`TestRetrievalQualitySuite` 从测试临时
目录读数据"那个全局污染坑的**根因**。P1-2 当时是绕过（不触发初始化），R2 是正面治理。

**方案**：`withTempKnowledge(t)` helper 同时 save/restore **两个**全局态：
1. `knowledgeDir`
2. `currentStorage`（在 setup 时置 nil 强制重新初始化；cleanup 时关闭测试存储并还原旧值）

cleanup 里只在 `currentStorage != oldStorage` 时才 `Close()`，避免误关原始存储。
`knowledge_graph_test.go` 同步简化（-25 行），不再各自手搓临时目录逻辑。

### S1：knowledge.go 按职责拆分（commit `778f892`）

**问题**：`knowledge.go` 3578 行 / 93 函数，单文件承载类型、存储、CRUD、检索管道、
embedding、链接、维护、分析、工具注册——典型上帝文件，任何改动都要在巨文件里翻找。

**方法论（可复用，关键是"安全地拆"）**：
包内重构（都是 `package tools`）的本质是——**移动函数不改行为、不改调用点**（同包互访）。
唯一的失败模式只有三种，且全部被工具链拦截：
1. 语法错误 → gofmt 拦
2. 未用/缺失 import → `go build` 拦
3. 重复声明 → `go build` 拦

所以只要 build 绿就等于正确。具体手法（脚本 `/tmp/split2.py`）：
- 从**单一权威快照**按 1-indexed 行号区间切片（避免边切边移动导致行号漂移）
- 每个区间带一个**内容断言**（边界行必须含某 marker 子串），防止行号错位
- 断言区间**严格递增 + 完全平铺**（`total == N - 20`，20 是 package+import 头）
- 对每个目标文件**正则扫描搬过去的函数体**（`\b<alias>\.`）自动推导所需 import，
  连 core 自己的 import 块也重新生成——这样被搬走的代码不会留下未用 import
