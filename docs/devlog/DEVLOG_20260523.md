## code_deep_analyze 工作流改造：去本地模型化

### 背景

`code_deep_analyze.yaml` 的 `analyze_files` 步骤原使用 `provider: local`（Qwen3.6 本地模型）做逐文件分析。本地模型速度慢且需要 GPU。

### 改动

将 `analyze_files` 步骤的 `provider` 从 `local` 改为 `deepseek`，全部走 API：

| 位置 | 改前 | 改后 |
|------|------|------|
| provider | `local` | `deepseek` |
| 成本估算 | DeepSeek 2次 + Qwen 免费 | DeepSeek 3次 ≈ ¥0.08 |
| 阶段标题 | Qwen3.6 并发批量逐文件深读 | DeepSeek 逐文件分析 |
| 标签 | 双模型分析 | 全 API 分析 |

### 验证

`go build ./...` 通过后，用 beishan-core 自身的 `code_deep_analyze` 工作流做了端到端测试。8 步全部成功，`analyze_files` 改用 DeepSeek API 后输出正确 JSON 格式分析结果。

## Go-DSL 工作流引擎设计

### 背景

L3 插件当前**数量少、质量参差**，每个插件手写 OnMessage switch-case。L4 编排器（legal_review）也靠手写 `kernel.Call` 链。需要一个**编译时安全的 Go-DSL 工作流系统**，降低创建新 L3 插件的门槛，提升质量。

### 定位

与现有 YAML 引擎（`internal/workflow/engine.go`）协同：

| 维度 | YAML 引擎 | Go-DSL 引擎 |
|------|-----------|-------------|
| 使用者 | AI / 最终用户 | Go 开发者 |
| 定义方式 | YAML 文件 | Go struct |
| 类型安全 | 运行时检查 | 编译时检查 |
| 变更频率 | 高频（AI 可修改） | 低频（核心管道） |
| 定制逻辑 | 仅 plugin call | Before/AfterExecute 钩子 |
| 定位 | L4 高层编排 | L3/L4 静态硬化链 |

### 三条架构约束

1. **编译时校验 Tool 注册表** — `NewGoToolPlugin` 构造时检查所有 Tool 在 `tools.GetToolSchema()` 中已注册，未注册 panic
2. **零冗余校验** — StepTool 通过 `kernel.Call` 走标准消息路由，Go-DSL 不做参数校验，信任 L3 硬化层
3. **Before/AfterExecute 不做 I/O** — 纯数据变换约定（改写查询、过滤结果），非代码强制但文档明确

### 实现结果

已完成 Go-DSL 工作流引擎的完整实现：

| 文件 | 操作 | 行数 | 说明 |
|------|------|------|------|
| `internal/workflow/types.go` | 修改 | +80 | 新增 GoStep、GoWorkflow、StepStatus、ErrorStrategy 等类型 |
| `internal/workflow/gods_executor.go` | 新增 | ~290 | GoExecutor + StateStore + GoWorkflowPlugin + NewGoToolPlugin |
| `legal_review_go_dsl.go` | 新增 | ~80 | legal_review 的 Go-DSL 声明式版本（等价 YAML 工作流） |
| `main.go` | 修改 | +21 | toolHost + ensureTool 注册表 + Go-DSL 插件注册 |

**架构约束落实：**
- 规则 1：`NewGoWorkflowPlugin` 构造时调用 `validateGoStep` 递归检查所有 `GoStep.Tool`，未注册 panic
- 规则 2：`GoStepTool` 执行器通过 `kernel.Call` 经标准消息路由 → L3 `OnMessage` → `ValidateAndExecute`，Go-DSL 不做参数校验
- 规则 3：`GoStep` 中 `BeforeExecute`/`AfterExecute` 字段含注释"约定不做 I/O"

**未迁移保留原实现的插件（各有特殊逻辑）：**
- search_plugin — 含 LLM query rewrite（rewriteWebQuery）+ think_plugin 摘要总结
- memory_plugin — 动态 `tools.HasTool` 分发
- write_plugin — 不同 Type 有不同的返回行为（fire-and-forget vs 返回结果）

### 关键设计决策

- **与 YAML 引擎共存**：Go-DSL 放在 `internal/workflow/` 同一包，共享 `StepResult`/`WorkflowResult` 类型，不创造第二套
- **StateStore 结构化取值**：`Get("step_id.field.nested")` 点号路径取值，复用 `extractFieldFromValue`，不走字符串替换
- **toolHost 映射**：tool 名 → 宿主插件名（如 `web_search` → `search_plugin`），在 main.go 中集中配置
