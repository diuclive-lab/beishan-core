### 事件机制设计

方案 D：JSONL 持久化 + 单层 subscriber。事件即持久化，不分离。

```
observatory.PublishEvent(event)
  ├─ 写入 events_YYYYMMDD.jsonl（审计）
  └─ 同步通知所有匹配 subscriber

observatory.Subscribe("agent.*", handler)
  └─ 按事件名前缀匹配（"agent."匹配agent.spawn/complete/failed）
```

**事件类型**：
- `agent.spawn` — 子智能体启动（携带 agent_id + task_prompt）
- `agent.complete` — 子智能体完成（携带完整消息快照）
- `agent.failed` — 子智能体失败

**持久化**：`agent.complete` subscriber 将对话写入 `eval/run/conversations/{agent_id}_{timestamp}.json`，供回溯使用。

### 会话持久化示例

```json
{
  "agent_id": "researcher",
  "iterations": 1,
  "output": "24",
  "messages": [
    {"role": "assistant", "content": "24"}
  ]
}
```

### 核心认知

1. **"设计决策"是甩锅最常用的借口**。今天拆的三条雷里，评分归一化和 SESSION_EXPIRED 根本不是什么设计决策——就是没想到。诚实面对这一点比找一个漂亮的理由更重要。

2. **事件机制补齐了一个架构级短板**。之前所有"通知类"需求都被阻塞（embedding 进度、failover 通知、子智能体进度）。现在有了 `PublishEvent` + `Subscribe`，这类需求都有了基础。

3. **持久化和事件是一体两面**。不是先做事件再做持久化，而是事件本身就是持久化——每条事件都是不可变的状态记录。需要回溯时直接查事件日志即可。

## 架构新增：通道层 + 记忆存储余量设计（2026-05-25 下）

### 多通道消息层（`internal/channels/`）

余量接口设计。当前无实现通道，仅定义接口和注册表。

```go
type Channel interface {
    Name() string
    Send(msg ChannelMessage) error
    Connect() error
    Close() error
}

type Manager struct {
    channels map[string]Channel  // 线程安全注册表
}
```

**设计要点：**
- `ChannelMessage` 统一结构（channel/recipient/subject/body/html）
- `Manager.SendAll()` 支持向全部通道广播
- 注册重复名称时返回错误，不静默覆盖
- 通道实现由 Mac B 参考 66 项目（FangLab，阿里云轻量服务器方案）完成

### 记忆存储接口（`internal/memory/`）

余量接口 + fileStore 当前实现。

```go
type MemoryStore interface {
    Get(key string) (*MemoryItem, error)
    Set(item MemoryItem) error
    Delete(key string) error
    Search(query string, limit int) ([]MemoryItem, error)
    List() ([]string, error)
}
```

**FileStore 实现：**
- 每条记忆一个 JSON 文件
- 基础关键词搜索（大小写不敏感），TTL 过期支持
- `ErrNotFound` 哨兵错误

**预留实现（不完整吸收 OpenClaw memory-core）：**
- `SQLiteStore`：结构化记忆，事务支持
- `EmbeddingStore`：向量检索，语义搜索

### 设计原则

这两层都是余量设计（margin-of-safety），不是完整实现：
- **不增加当前复杂度** — 接口干净，fileStore/无通道也能正常运作
- **不形成技术债务** — 零外部依赖
- **扩展时无需重写** — 加新通道/新存储只需实现接口

### 设计来源

Qwen Code `/review` skill（DESIGN.md）的 9 智能体并行审查模式：
- 6 个审查维度：逻辑正确性 / 安全漏洞 / 代码质量 / 性能 / 测试覆盖率 / 非定向
- 3 个非定向角色：攻击者视角 / 3am 值班工程师 / 长期维护者
- 批量验证：单智能体接收 9 路结果，逐条确认
- 迭代逆向审计：最多 3 轮（clean 提前终止）

### 实现

`workflows/code_review_9x.yaml` — 7 步工作流：

```
get_diff（git diff 或读文件）
  → parallel_review（9 路并行，type:parallel）
  → verify（批量验证汇总）
  → reverse_audit_r1-r3（迭代审计，最多 3 轮）
  → report（写文件）
```

**技术踩坑：** YAML 的 `default` 字段是 bool 类型。`- default: reverse_audit_r2` 会被解析为 `default: "reverse_audit_r2"`，但 struct 定义是 `Default bool`，导致 yaml unmarshal 失败。正确写法为 `- default: true` + `goto: reverse_audit_r2` 两行。

### 实测

审查对象：`kernel/router.go`（~220 行）

| 结果 | 值 |
|------|------|
| 总耗时 | **37 秒** |
| 步骤数 | 5（r2/r3 因 clean 跳过） |
| 总发现 | 11（含 3 个误报排除后 8 个有效） |
| 严重级分布 | 1 critical + 5 high + 2 medium + 3 low |
| 逆向审计 | 第 1 轮 clean，提前终止 |

### 发现的真实问题

审查报告确认了工作流自身的缺陷：

| 严重级 | 问题 | 影响 |
|:------:|------|------|
| Critical | reverse_audit 无限循环风险 | default 分支缺乏轮次上限检查 |
| High | 逻辑绕过 | default:true+goto 绕过状态判断 |
| High | 测试覆盖率全面缺失 | 边界值/分支/API 均未覆盖 |
| Medium | r3 未定义 next | 非 clean 状态静默结束 |
| Medium | 跳转监控缺失 | 异常难以定位 |
| Low | 重复代码 | next 逻辑复制粘贴 |

### 意义

这是第一次真正验证了**硬化层 + YAML 工作流 + 外部工具**的完整闭环：
1. 工作流引擎调 `terminal_plugin` 获取 diff（受硬化层监督）
2. 9 路 `think_plugin` 并行调 DeepSeek 审查（通过 ValidateAndExecute）
3. `write_plugin` 写报告（受硬化层校验）
4. 审查报告返回到会话（通过 kernel 回程路由）

**37 秒完成 9 路 LLM 调用**也验证了工作流引擎的并行 goroutine 模式在实际负载下的可用性。
