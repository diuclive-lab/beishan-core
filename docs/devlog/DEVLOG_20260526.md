### 四、工作流引擎 Bug 修复

**问题：** `resolveJSONValue` 递归无深度限制，处理 `code_tree` 大 JSON 输出时栈溢出。

**修复：** 新增 `resolveJSONValueDepth(raw, depth int)`，depth > 10 时截断返回原始数据。

### 执行细节

**创建 docs/DATA_FLOW.md**（全新文件，~140 行）

4 条已验证路径刻画了系统真实走通的数据流：
- 路径 A：HTTP POST `/api/chat` → kernel → Router(DeepSeek) → plugin → HTTP response
- 路径 B：think_plugin 内部带 RAG 检索 + 查询改写 + 质量门禁的完整链路
- 路径 C：glue IPC 子进程通信（stdin/stdout JSON 行协议 + 30s 超时）
- 路径 D：右花调用（HTTP POST dispatch → adapter → 目标服务）

4 条断路路径记录了已知问题：
- 路径 E（SessionHandler=nil，高优先级）：异步回程消息静默丢失
- 路径 F（parseToolSuggestions 孤岛，中优先级）：LLM 无法请求工具调用
- 路径 G（observatory 无消费者，中优先级）：系统不可观测
- 路径 H（readEvent 被遮蔽，低优先级）：子进程 event 推送不达

**注入 CLAUDE.md**（+97 行）

在 AI Guardrails 之后新增完整"集成纪律"章节。关键设计决策：

- 对话开始时必须有强制检查流程（`scripts/integration_check.sh` 输出 + `DATA_FLOW.md` 路径状态）
- 声明完成前必须输出 `---INTEGRATION_PROOF---` 结构块，不可跳过
- 新增"被允许说的话/不被允许说的话"区分，防止 AI 形式合规但实质空洞
- 占位符规则禁止空接口/空结构体伪装成实现，必须显式 `UNIMPLEMENTED` 注释
- kernel 冻结的硬性规定从设计原则提升到 CLAUDE.md 的可执行规则
- "你不是连续工作的"节直接针对 AI 无跨会话记忆的固有缺陷

**合并 DESIGN_PRINCIPLES.md**（+148 行 + 后续 +10/-4 行）

"集成纪律"章节从设计哲学角度深入分析了三类死代码的成因：
- 类型一（孤岛函数）：以 `parseToolSuggestions` 为案例——函数完整但调用点不存在
- 类型二（占位符伪装）：以 `internal/channels/` 为案例——结构完整但无导入
- 类型三（断路模块）：以 `SessionHandler = nil` 为案例——逻辑完整但上游未注入

这些案例和分析来自桌面版的独立诊断，与项目真实代码精确对应。

### pre-commit 首次拦截事件

这是最值得记录的一刻：**pre-commit hook 在第一次提交时就拦截了**。

### 关于集成纪律的反思

集成纪律框架的注入不只是"加了一堆文档"。它的结构有明确的层次：

```
CLAUDE.md "集成纪律"                ← AI 的执行规范（即时约束）
  └─ 你的默认假设                     ← 认知基模（"怀疑上一次"）
  └─ 对话开始强制检查                 ← 触发时机（什么时候检查）
  └─ INTEGRATION_PROOF 输出格式       ← 完成标准（什么叫做完）
  └─ 被允许说的话                     ← 行为边界（能怎么说）

DESIGN_PRINCIPLES.md "集成纪律"       ← 设计原则（为什么这么做）
  └─ 问题的根源 / 三类死代码           ← 成因诊断
  └─ 完成的定义 / 占位符标准           ← 质量标准
  └─ DATA_FLOW.md / integration_check  ← 工具要求

scripts/integration_check.sh          ← 机器检查（外部验证）
.git/hooks/pre-commit                 ← 阻断机制（最后防线）
```

这是一个从"原则 → 规范 → 工具 → 阻断"的完整闭环。不是靠 AI 自觉，而是靠外部工具在代码层面强制验证。

### 与现有治理框架的关系

`docs/ABSORPTION_GOVERNANCE.md` 治理框架负责**外部能力吸收的质量控制**（证据等级 E1-E4、吸收等级 L0-L4、风险分类），而集成纪律框架负责**底座自身开发过程的完整性控制**（新增即连通、占位符诚实、完成定义）。两者互补不冲突：

| 框架 | 控制对象 | 核心问题 | 验证手段 |
|------|---------|---------|---------|
| 吸收治理 | 右花→底座的能力吸收 | 吸收是否充分 | 证据等级 + 缺口分析 |
| 集成纪律 | AI 在底座上的开发 | 代码是否真的集成 | INTEGRATION_PROOF + pre-commit |

### 工作流跳步 bug 修复

`project_analyze` 和 `code_deep_analyze` 两个工作流存在 `static_metrics` 跳过中间步骤的问题。

- ✅ 最终方案：保持引擎不变，修复 YAML 中 `next:` 链

从 FangLab 吸收了 `internal/tokenbudget/` 包并尝试集成到 worklow 引擎和 llm 层。

**放弃原因（实测数据）：**

- `internal/tokenbudget/` 已删除
