## R4：门禁硬化——把「看着全实则只挡 test/vet」改成真门禁

### 病灶（坐实记分卡 R4）

`eval/scripts/core_gate.sh` 列了 10 项检查，**看着很全**。逐行读后坐实两层 theater：

1. **结构层 theater**：除 `go test` / `go vet` 外的 8 项，默认全是
   `else echo "⚠️"; $STRICT && FAILED=1`——即**不加 `--strict` 就只提示、不阻断**。
   默认跑 `core_gate.sh` 实际只有 test + vet 两项能挡住提交。
2. **判定方式 theater**：用 `bash 子脚本 | tail -1 | grep -q "关键词"` 判过。
   `check_security_model` 的成功词是 `一致`，失败词是 `不一致`——**`一致` 是 `不一致` 的子串**，
   `grep -q "一致"` 对失败行也命中 → 失败被误判为通过。

更深一层：**两个被门禁调用的子检查自身也是 theater**：
- `check_security_model.sh`：4 项 grep 全写成 `... && echo ✅ || echo ❌`，但 `❌` 分支
  **从不设 `FAILED`** → 脚本永远 `exit 0`、永远打印 `✅ 安全模型一致`。门禁 grep 它的末行永远过。
- `scan_boundary.sh`：实际**正在失败**（exit 1）——`plugins/workflow_plugin.go`（引擎读自身
  workflow 目录）和 `plugins/knowledge_calibration.go`（校准 JSONL 内部固定路径）两处 L4 直接
  文件系统 I/O **未登记**进边界债务册，扫描判为「含未知违规」。因为这项一直 ⚠️ 不阻断，债务静默累积。

### 设计：诚实分层，不是「全部变硬」

「把 8 项全 promote 成 blocking」是错的——`core-eval`（跑 bench、要 LLM）、`rightflower smoke`
（要右花 adapter 在线）、`core-health`（探 provider 可达性）都依赖**活服务/网络**，硬化它们会让
门禁在没有服务的开发机上**必然 flaky**（正是 hermetic 门禁要避免的，Task L 教训）。

故按**是否离线确定性**分两档：

| 档 | 判据 | 成员 | 行为 |
|----|------|------|------|
| BLOCKING | 离线、确定性（grep/静态扫描） | go test / vet / boundary / workflow-v2.5 / docs / workspace / security（7 项） | 恒阻断，无视 flag |
| ADVISORY | 需活服务/网络，可能 flaky | core-health / core-eval / rightflower（3 项） | 默认只提示；`--strict` 才阻断 |

落实前先逐个确认 7 个 BLOCKING 成员**当前都能过**（否则硬化即锁死所有提交）：4 个直接绿，
`security` 修 FAILED 后绿，`boundary` 补登记 D04/D05 后绿。

### 改法

1. **判定改退出码**：`core_gate` 改用子脚本**退出码**（`if bash script; then ✅ else ❌ FAILED=1`），
   不再 grep 末行关键词——根除 `一致`/`不一致` 子串误判。
2. **修 `check_security_model.sh`**：4 个 `❌` 分支补 `FAILED=1`（4 目标现存，仍绿，但已为真）。
3. **补登记 D04/D05**（`docs/reports/boundary_debt_register.md`）：两处均为**内部固定路径** I/O
   （非用户输入、非路径穿越），同 D02/D03 性质；`scan_boundary.sh` 的 KNOWN pattern 同步加
   `workflow_plugin|knowledge_calibration` → 扫描转「仅已知债务 D01-D05」通过。
4. **重写 `core_gate.sh`**：`block()` / `advisory()` 两个 helper，BLOCKING 恒阻断、ADVISORY 受
   `--strict` 控制；`--quick` 仍只跑 go test。

### 验证

```
全量 core_gate.sh：
  [blocking] go test/vet/boundary/workflow-v2.5/docs/workspace/security → 7×✅
  [advisory] core-health ⚠️(warn，开发机无 key) / core-eval ✅ / rightflower ✅
  exit 0
负向单测（坐实不再是 theater）：
  block "failing" false        → FAILED=1, exit 1   ✅ BLOCKING 真能挡
  advisory(STRICT=false) false → FAILED=0, exit 0   ✅ 默认不阻断
  advisory(STRICT=true)  false → FAILED=1, exit 1   ✅ --strict 升为阻断
```

### 教训

- **门禁的可信度 = 最弱一环**。一个「看着全」的门禁比「明确只挡 2 项」更危险——它给虚假安全感。
- **两种 theater 都要堵**：① 结构层默认降级（8 项 ⚠️ 不阻断）；② 检查层自身永不失败
  （`check_security_model` 不设 FAILED / `scan_boundary` 失败被 ⚠️ 掩盖）。
- **退出码 > 末行关键词**：关键词判定有子串陷阱（`一致`⊂`不一致`），退出码是脚本的一等公民信号。
- **硬化前先让被硬化项变绿**：promote 一个正在失败的检查到 blocking = 锁死所有提交。先登记/修复，再硬化。
- **`core_gate.sh` 仍未接入 pre-commit**（当前 hook 用 `integration_check.sh`）。本次只让它**变诚实**，
  未改其接入面——是否纳入 pre-commit 是单独决策（接入即每次提交都跑 7 项 + 3 advisory，有延迟成本）。

  `scan_boundary.sh`（KNOWN pattern +2 文件）、`docs/reports/boundary_debt_register.md`（+D04/D05）、

## 浏览器/搜索可靠性——止住「裸抓搜索引擎被反爬 + 静默把垃圾当成功」

用户实测：智能体"用浏览器搜索"时出问题。让我排查 + 判断更换还是迭代。

### 排查（全程实测取证，不猜）

两套机制并存：`web_search`/`web_fetch`（API：Tavily/Firecrawl/Brave，**付费 key 门禁**）和
`browser_navigate`（裸 HTTP，**永远在线**，bot UA，不渲染 JS）。实测：

| 路径 | 实测结果 |
|------|---------|
| `browser_navigate` 抓 DDG/Bing（bot UA `HermesAgent/1.0`） | **302→同意/拦截页，~140 字节**，无结果 |
| `html.duckduckgo.com`（proper Chrome UA） | **HTTP 202 "anomaly" 反爬挑战页，0 结果** |
| Bing keyless（Chrome UA） | HTTP 200 但仅 ~1 条可解析锚点（正则脆弱） |
| Tavily（付费 key） | ✅ 稳定返回真实结果 |

**根因不在"浏览器"本身，而在"用浏览器去搜索"这条路整条坏**：①`fetchAndExtract` 用 bot UA + 不渲染
JS + 只在 ≥400 报错→把 302/202 挑战页当成功**返回垃圾**；②`web_search` 即使 0 结果也回
`Success:true`（静默失败）；③`web_search` 被付费 key 门禁挡住，没 key 时整个工具消失，智能体只能
退回坏掉的 `browser_navigate`；④`browser_navigate` **绕过了** `isSafeURL`(SSRF)+`containsSecret`
（`web_fetch` 有、它没有）——能打内网 IP；⑤`browser_scroll`/`browser_back` 是假 no-op。

### 判断：迭代，不更换

免费爬搜索引擎的时代已经结束（DDG 都上 202 反爬了），换一个"浏览器工具"也躲不开反爬。架构其实
是对的（`web_search`+API、`web_render` Playwright、SSRF/密钥检查都在），是**接线接错**。用户拍板
「Tier-1 硬化 + 接入 Tavily」，并给了 Tavily key（另提到 FangLab 有「无头浏览器登录 DeepSeek 网页版
用其免费搜索」方案——但不在本仓库，作为右花集成备选）。

### 改法（Tier-1 止血，code 层）

- `web.go`：① `webSearchHandler` 0 结果→`Success:false`+`Error`；② `fetchAndExtract` UA 换真实
  Chrome；③ 检测反爬（202/204 + `unusual traffic`/`人机验证`/`/cdn-cgi/challenge` 等强信号，**不收
  `captcha` 单词**避免误伤）→报错而非返垃圾。
- `browser.go`：① 补 `isSafeURL`+`containsSecret`（与 `web_fetch` 对齐，止 SSRF/外泄）；
  ② `isSearchEngineURL` 识别搜索 URL→拒绝并指向 `web_search`；③ scroll/back 诚实化。
- 接入 Tavily：dev `.env` 配 `TAVILY_API_KEY`（gitignored，**未提交**）；实测 `web_search`→Tavily 端到端 OK。

### 验证

- 4 测试：`isSearchEngineURL` / `looksLikeAntiBot`（含不误伤断言）/ `browser_navigate` 私有 IP
  SSRF 拦截（hermetic，前缀短路不走 DNS）/ `web_search`→Tavily 集成（有 key 才跑，`verify.sh` 去
  key 故跳过——已把 3 个搜索 key 加进 verify 的 unset 列表）。
- `bash scripts/verify.sh` 全绿；新代码 gofmt-clean。

### 关键约束 & 遗留

- **现网 daemon 要让搜索真生效，需在其启动环境补 `TAVILY_API_KEY`**——包装脚本含密钥、不由本仓库
  管，由用户加（本会话遵守密钥护栏，未碰包装脚本）。在补 key 之前，无 key 的 daemon 上 `web_search`
  不注册 + `browser_navigate` 拒绝搜索 = 搜不了但**报错明确**（胜过静默垃圾）。
- 遗留：`browser_click` 同行 `href` 字符串匹配仍脆弱，且经 navigate 改造后**相对链接会被 SSRF 拦**
  （`isSafeURL` 对无 host 的相对 URL 返回 false）——browser_click 本就基本失效，记入 KNOWN_LIMITATIONS。
- 发现但未动（避免扩张）：`web_fetch`/`web_extract` 也被同一付费 key 门禁挡住，但抓 URL 根本不需要
  搜索 key——这道门禁是错的，留作后续。

## DeepSeek 网页版搜索：CDP-over-pipe 原生 Go 移植（吸收 FangLab）

承上：Tavily 接好后，用户要把 FangLab（beishan 的父项目）的「无头浏览器登录 DeepSeek 网页版、用其
免费联网搜索」也吸收进来，落地为**原生 Go**，并捕获 DeepSeek 的「已思考」（深度推理时有惊喜）。

### 机制（读懂 FangLab 源码）

`FangLab/scripts/mcp_runtime/browser_mcp_server.py` 的 `search_via_deepseek`：**不爬搜索引擎 SERP**，
而是把登录态的 DeepSeek 网页版当成「会联网搜索的 LLM」——填一段提示词让它只回包在标记里的结构化
JSON，轮询正文抽取。传输用 Playwright(Python) over CDP。

### 关键判断：路线选择

- 发现 **beishan 其实不是零依赖**：`go.mod` 已依赖 `playwright-go`（`web_render.go` 在用）+ pdf/yaml/
  lute/chroma 等。所以"CDP-over-pipe 为了零依赖"的前提不成立。
- 用户仍选**手写 CDP-over-pipe**（不深化 playwright 依赖、用系统 Chrome），并提出 north-star：
  参考 **Servo** 把浏览器引擎内嵌进智能体根茎，让浏览器成为智能体的一部分。
- CDP-over-pipe 正是这条路的**第一步**：beishan `exec` 启动 Chrome 加 `--remote-debugging-pipe`，
  CDP 走 fd 3/4 的 null 分隔 JSON，**进程由 beishan 拥有、私有管道通信**，不经 websocket、不用
  playwright 库、不起外部 driver。

### 实现

- `internal/tools/cdp.go`：极小 CDP-over-pipe 客户端（`newCDPConn` 起 Chrome + ExtraFiles=[r3,w4] 映射
  fd 3/4；`send` id 关联请求/响应；`attachPage` createTarget+attachToTarget(flatten)；`evalString`
  跑 JS；`insertText`/`pressEnter` 走 CDP Input 域，对 React 输入框友好）。
- `internal/tools/deepseek_web.go`：navigate→登录检测→`ensureDeepseekToggleOn`(智能搜索/深度思考)→
  聚焦输入→`insertText`(提示词)→等 React→`pressEnter`→轮询正文抽 `<BEISHAN_JSON>`→规范化 + 抽「已思考」。
- `cmd/deepseek-login`：一次性「有头」登录（持久化 profile，headless 复用）。

### 分步验证（每步实测取证）

1. **传输**：起 headless Chrome 跑 `navigator.userAgent` → 拿回 `HeadlessChrome/147`（5.25s）。✅
2. **navigate + 登录检测**：未登录 profile → 正确返回「需一次性登录」指引（5.8s 不挂）。✅
3. **用户登录后端到端**：首跑 **90s 超时**——诊断（逐步 dump）发现输入/提交/响应**都正常**，问题在
   `insertText` 后**立即** `pressEnter`，React 没登记输入就回车 → 空发。
4. **登录后 3 个真实坑**（对着真页面调）：
   - **插入→回车之间等 ~900ms**（根因，已修）。
   - 开关叫「**智能搜索/深度思考**」（非"联网搜索"）；用 `aria-pressed` 判状态、只在关时点开。
   - 开「智能搜索」后 DeepSeek 把**引用角标/换行注入 JSON 字符串值**→非法 JSON；`extractMarkedJSON`
     加「内部空白折空格重试」修复。「已思考」锚点是「已思考（用时」而非"已深度思考"。
5. **修后端到端**：`success=true / 4 条真实 2026 结果 / 已思考 2105 字节 / 45s`。✅

### 教训

- **scrape 聊天 UI 的脆弱性是真实的**：输入时序（React 异步）、开关命名/状态、富文本注入（引用角标进
  JSON）——这些只有**对着真实登录页逐步 dump** 才能发现，纯靠读参考源码会漏。「分步验证、每步取证」
  是这类活的唯一靠谱姿势。
- **CDP Input 域 > JS 注入**：`Input.insertText`/`dispatchKeyEvent` 模拟真实输入，对 React 受控组件
  友好，省掉一堆「设 value 不触发 onChange」的坑。
- **靠文本锚点、不靠哈希类名**：DeepSeek 类名是哈希的（`_6dbc175`），「已思考」用文本锚点抽，稳。
- north-star（内嵌 Servo）见 `docs/SERVO_BROWSER_NORTHSTAR.md`。
