## R4：门禁硬化——把「看着全实则只挡 test/vet」改成真门禁

### 病灶（坐实记分卡 R4）

`eval/scripts/core_gate.sh` 列了 10 项检查，**看着很全**。逐行读后坐实两层 theater：

1. **结构层 theater**：除 `go test` / `go vet` 外的 8 项，默认全是
   `else echo "⚠️"; $STRICT && FAILED=1`——即**不加 `--strict` 就只提示、不阻断**。
   默认跑 `core_gate.sh` 实际只有 test + vet 两项能挡住提交。
2. **判定方式 theater**：用 `bash 子脚本 | tail -1 | grep -q "关键词"` 判过。
   `check_security_model` 的成功词是 `一致`，失败词是 `不一致`——**`一致` 是 `不一致` 的子串**，
   `grep -q "一致"` 对失败行也命中 → 失败被误判为通过。

更深一层：**两个被门禁调用的子检查自身也是 theater**：
- `check_security_model.sh`：4 项 grep 全写成 `... && echo ✅ || echo ❌`，但 `❌` 分支
  **从不设 `FAILED`** → 脚本永远 `exit 0`、永远打印 `✅ 安全模型一致`。门禁 grep 它的末行永远过。
- `scan_boundary.sh`：实际**正在失败**（exit 1）——`plugins/workflow_plugin.go`（引擎读自身
  workflow 目录）和 `plugins/knowledge_calibration.go`（校准 JSONL 内部固定路径）两处 L4 直接
  文件系统 I/O **未登记**进边界债务册，扫描判为「含未知违规」。因为这项一直 ⚠️ 不阻断，债务静默累积。

### 设计：诚实分层，不是「全部变硬」

「把 8 项全 promote 成 blocking」是错的——`core-eval`（跑 bench、要 LLM）、`rightflower smoke`
（要右花 adapter 在线）、`core-health`（探 provider 可达性）都依赖**活服务/网络**，硬化它们会让
门禁在没有服务的开发机上**必然 flaky**（正是 hermetic 门禁要避免的，Task L 教训）。

故按**是否离线确定性**分两档：

| 档 | 判据 | 成员 | 行为 |
|----|------|------|------|
| BLOCKING | 离线、确定性（grep/静态扫描） | go test / vet / boundary / workflow-v2.5 / docs / workspace / security（7 项） | 恒阻断，无视 flag |
| ADVISORY | 需活服务/网络，可能 flaky | core-health / core-eval / rightflower（3 项） | 默认只提示；`--strict` 才阻断 |

落实前先逐个确认 7 个 BLOCKING 成员**当前都能过**（否则硬化即锁死所有提交）：4 个直接绿，
`security` 修 FAILED 后绿，`boundary` 补登记 D04/D05 后绿。

### 改法

1. **判定改退出码**：`core_gate` 改用子脚本**退出码**（`if bash script; then ✅ else ❌ FAILED=1`），
   不再 grep 末行关键词——根除 `一致`/`不一致` 子串误判。
2. **修 `check_security_model.sh`**：4 个 `❌` 分支补 `FAILED=1`（4 目标现存，仍绿，但已为真）。
3. **补登记 D04/D05**（`docs/reports/boundary_debt_register.md`）：两处均为**内部固定路径** I/O
   （非用户输入、非路径穿越），同 D02/D03 性质；`scan_boundary.sh` 的 KNOWN pattern 同步加
   `workflow_plugin|knowledge_calibration` → 扫描转「仅已知债务 D01-D05」通过。
4. **重写 `core_gate.sh`**：`block()` / `advisory()` 两个 helper，BLOCKING 恒阻断、ADVISORY 受
   `--strict` 控制；`--quick` 仍只跑 go test。

### 验证

```
全量 core_gate.sh：
  [blocking] go test/vet/boundary/workflow-v2.5/docs/workspace/security → 7×✅
  [advisory] core-health ⚠️(warn，开发机无 key) / core-eval ✅ / rightflower ✅
  exit 0
负向单测（坐实不再是 theater）：
  block "failing" false        → FAILED=1, exit 1   ✅ BLOCKING 真能挡
  advisory(STRICT=false) false → FAILED=0, exit 0   ✅ 默认不阻断
  advisory(STRICT=true)  false → FAILED=1, exit 1   ✅ --strict 升为阻断
```

### 教训

- **门禁的可信度 = 最弱一环**。一个「看着全」的门禁比「明确只挡 2 项」更危险——它给虚假安全感。
- **两种 theater 都要堵**：① 结构层默认降级（8 项 ⚠️ 不阻断）；② 检查层自身永不失败
  （`check_security_model` 不设 FAILED / `scan_boundary` 失败被 ⚠️ 掩盖）。
- **退出码 > 末行关键词**：关键词判定有子串陷阱（`一致`⊂`不一致`），退出码是脚本的一等公民信号。
- **硬化前先让被硬化项变绿**：promote 一个正在失败的检查到 blocking = 锁死所有提交。先登记/修复，再硬化。
- **`core_gate.sh` 仍未接入 pre-commit**（当前 hook 用 `integration_check.sh`）。本次只让它**变诚实**，
  未改其接入面——是否纳入 pre-commit 是单独决策（接入即每次提交都跑 7 项 + 3 advisory，有延迟成本）。
