### 3. Session 读取走正规路径

`loadRecentSessionMessages` 原来：


### 4. Session 摘要自动生成（接通断路）

## 设计原则沉淀

本次修复确立了几条可复用原则，已写入知识库：

1. **单一词表原则**：关键词只在 `intent_keywords.go` 定义，其他文件只引用。修改词义只改一处。
2. **OnMessage 分发原则**：OnMessage 是纯路由，不含任何业务逻辑。状态机和自然语言严格分叉。
3. **Session 读取原则**：必须通过 `tools.SessionGet`，不直接操作文件路径。
4. **摘要生成原则**：每次对话结束后异步生成 session 摘要，确保跨会话检索可用。

### 一、llmguard 行为契约层

**创建 `internal/llmguard/`（8 个文件）**，LLM 调用的框架级约束中间件。

#### 设计动机

原本 V25 规则散落在每个 workflow YAML 的提示词里，plugin 直接调 `llm.*` 时没有任何约束兜底。llmguard 把"LLM 应该怎么表现"从文本约束升级为可编程的契约对象。

#### 核心洞察：强制性按维度分配

| 维度 | 强制策略 |
|------|---------|
| 结构（output schema） | **层 2 满强度** — 强制不影响内容质量 |
| 内容（reasoning quality） | **层 1+4 半强度** — 过强让模型变笨 |
| 事实（grounding） | **层 4 critique** — 强制查"有没有出处" |

#### 维度化 API

```go
ForStructure("json", "findings,risk_register", 1)  // 结构维度
ForContent()                                         // 内容维度（AntiLazy基线）
ForFacts()                                           // 事实维度（含Critique）
// Fluent组合：
ForStructure("json", "findings,risk_register", 1).WithContent().WithFacts()
```

#### 三层强度

| 层 | 机制 | 成本 |
|---|------|------|
| 1 | 注入 AntiLazy/RequireEvidence 基线到 system prompt | 零 |
| 2 | 输出格式校验（JSON schema）+ 违规重试 | 1 次重试成本 |
| 4 | Critique-Revise 二次调用自审改写 | 翻倍 |

#### 已接入调用点（8 处，全部覆盖）

| 调用方 | 契约 |

## 追加 5：embedding sidecar 常住化（5/27 第三轮）

### 问题

语义搜索的 embedding 服务依赖 nomic-embed llama-server 人工启动。机器重启后服务断掉，`EMBEDDING_ENDPOINT` 环境变量未在 `.zshrc` 中持久化，`embeddingEnabled()` 返回 false，L1 语义检索彻底失效无告警。

检索测试证实：纯 L0 关键词搜索对自然语言问题仅 4/10 命中，且 85 条知识（26%）的 summary 字段为纯噪声，关键词不可达。

### 方案：glue sidecar 管理器

不在 glue 中增加新的 IPC 协议，而是扩展 glue 子进程管理能力：

**glue/glue.go 变更：**
- 新增 `sidecar` 结构体 — 代表一个 HTTP 服务型辅助进程（非 IPC 插件）
- `GlueLayer.StartSidecar(name, command, args, port)` — 启动并跟踪
- 健康检查循环（`healthCheckLoop`）通过 TCP dial 检测 sidecar 存活
- 进程退出自动重启（`restartSidecar`）
- `ProcStatus()` 统一报告子进程 + sidecar 状态（前缀 `S:` 标识）
- `Shutdown()` 中终止所有 sidecar

**internal/tools/knowledge.go 变更：**
- 新增 `SetEmbeddingEndpoint(url)` — 程序化设置 embedding API 端点，覆盖环境变量
- 不依赖 `os.Setenv`，直接通过包级变量切换

**cmd/beishan/main.go 变更：**
- glue 启动后自动调用 `gl.StartSidecar("nomic-embed", "llama-server", ...)`
- 轮询 TCP :8092 等待就绪后调用 `tools.SetEmbeddingEndpoint()`
- 失败仅打日志，不阻止系统启动

## 未完成梳理和修复：AI 辅助开发文档误导问题

### 发现

检索诊断过程中，AI 错误地将两套独立的向量系统混为一谈（BOW 词袋 .embed.json 与 API 语义 entry.Embedding），根源在于关键架构知识只存在于历史 devlog 中，没有更新到 CLAUDE.md。

### 误导链条

1. `DEVLOG_20260521.md` 第 724 行清楚记录了两套 embedding 系统的区别，结论是"BOW 词袋向量足以支持本地语义检索，API embedding 是可选增强，当前不阻塞"
2. 但 CLAUDE.md 里的描述已过时且自相矛盾：
   - `EMBEDDING_ENDPOINT | No | — | Enables semantic search (not configured yet)`
   - `Embedding endpoint (was Qwen 27B — too heavy, use gemma-4-E4B next)`
   - `# Embedding model (llama.cpp on port 8090)`
3. AI 读到"not configured yet"时，不知道实际上有两套系统在运行
4. 于是 AI 看到 512 维 .embed.json 和 768 维 nomic-embed API，推断出"维度不匹配"
5. 但主搜索管道 `searchByEmbedding` 用的是 `entry.Embedding` 字段，根本不读 .embed.json

### 根因

**关键系统知识只存在于某一次对话的产物里，下一次对话的 AI 不知道。** 两套 embedding 的区别是 5 月 21 日那次开发中发现的，写进了 devlog，但没有更新到 CLAUDE.md 的 Key Files 表或 Architecture 段。之后每次新的 AI 对话，都不知道这个区别。
