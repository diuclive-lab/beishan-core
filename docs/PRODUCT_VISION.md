# beishan-core 产品定位与演进路线

> 创建：2026-05-27 | 最后更新：2026-05-28

## 一句话定位

一个跑在本地设备上的私有智能体，以个人知识库为核心，通过右花吸收机制持续进化，
挂载通用工具和工作流，支持远程控制，存储完全私有，越用越聪明。

> **注**：「存储私有」是当前阶段（Stage 1）的准确描述。阶段一使用云端 LLM 推理（DeepSeek API），
> 对话内容在推理时经过云端；「计算私有」是中期目标，待本地模型切换完成后成立。

---

## 核心机制：根茎生长

beishan-core 不是一个静态框架，是一个会生长的系统。

生长方式：右花探路 → 底座吸收 → 右花去探下一个领域。

右花（外部开源项目）通过协议接入后，不只是"可以调用"，而是作为学习对象。
吸收治理框架（ABSORPTION_GOVERNANCE.md）定义了五级吸收深度：

  L0 observe     — 只研究不实现
  L1 wrap        — 保留原实现，协议包装
  L2 partial     — 吸核心能力，外围保留外部
  L3 internalize — 完整内化，行为兼容
  L4 supersede   — 超越原设计

每次吸收都有证据等级（E1-E4）、风险评估、gap analysis。
不是"看到好的就抄"，是有纪律的学习。

这个机制保证底座和根茎不断壮大——
随时可以向最新最热门的开源项目学习，把验证过的能力内化为自己的硬化层能力。

## 三个阶段

### 短期：积累与校准（当前阶段）

系统基础设施刚完成梳理，产品第一次处于可正式使用状态。
这个阶段有两个并行目标：日常使用积累数据，同时保持吸收能力活跃。

**LLM 推理**：云端 API（DeepSeek），保证数据积累质量最高。
对话内容在推理时经过云端，这是阶段一的已知权衡——用质量换隐私。
阶段二切换本地模型后，推理也在本地，权衡消失。

日常使用积累：
- 用起来。不是测试，是当工具用。日常开发中用它记录决策、检索经验、执行操作。
- 验证检索质量。当前基线：L0 关键词 6/10、L1 语义检索 4/10（2026-05-27）。
  确认各层召回率随数据积累的改善趋势。
- 让校准系统跑起来。确认/拒绝入库产生校准数据，各类型的自动入库阈值逐步收敛。
- 观察高频操作模式，为后续切本地模型储备 preRoute 规则。

吸收能力保持活跃：
- 当前三个右花（OpenHuman、Hermes、OpenClaw）继续作为学习对象，
  已吸收的能力做 Step 3 验证，确认内化质量。
- 关注新的开源项目，随时可以作为新右花接入做 L0 研究。
- 吸收工作流（absorb_right_flower.yaml v2.5）已验证可用。

积累的数据：
- knowledge_calibration.jsonl — 知识分类精度
- EvidenceRouter EWMA 权重 — 高频意图的确定性路由强度
- 知识库条目 — 开发经验、决策记录、踩坑记录
- 用户画像 — 使用习惯和偏好
- session 摘要 — 跨会话历史检索的基础
- 吸收决策登记册 — 每次吸收的证据、等级、gap analysis

**阶段一结束的必要条件**（均需满足）：
1. 检索召回率可靠（L0 ≥ 7/10，L1 有稳定表现）
2. 校准系统各类型有足够样本（≥ 5 条，calibMinSamples 已设为 5）
3. 系统行为在日常使用中可预期，无频繁崩溃或路由错误

**可选成就**（不阻塞阶段推进，但加分）：
- 完成至少一次 L2+ 吸收闭环（从现有三个右花中选一个推进）


### 中期：切换本地模型 + 远程控制 + 持续吸收

积累了足够数据后，开始脱离云端依赖，同时保持吸收节奏。

本地模型切换：
- 评估本地模型在三个关键路径的表现：路由、检索改写、对话生成
- preRoute 覆盖面扩大，高频确定性操作直接走规则路由
- llmguard 的 Critique 按场景取舍
- Router prompt 适配本地模型特性（当前 prompt 为 DeepSeek 优化，需重新标定）

远程控制：
- iOS app 作为主要入口，通过 HTTP API 远程控制 Mac mini
- 高频操作通过 preRoute 直接路由，响应延迟控制在秒级
- **前置条件**：传输安全层需先就绪
  - TLS（HTTPS，或 tailscale 加密隧道）
  - 认证（Bearer token 或会话 token）
  - 网络穿透方案（tailscale / 固定 IP / frp）
  信任的前提是数据不出本地，但远程连接本身也需要传输加密——两者缺一不可。

持续吸收：
- Claude CLI 作为右花接入后，先做 L0/L1（研究+协议包装），
  再评估哪些编码能力值得 L2 内化到底座。
- 类 Obsidian 的开源知识管理工具（Logseq / Foam / Dendron 等）接入后，
  评估知识管理模式的吸收等级。目标是存储格式（markdown + YAML frontmatter），
  不绑定任何单一产品——符合右花协议"依赖是暂时的"原则。
- 持续关注新的开源 Agent 项目，作为右花做 L0 研究，
  有价值的模式按治理框架吸收。
- 每次吸收都强化底座的硬化层，让系统能力不依赖任何单一外部项目。

产品形态：本地运行的私有智能体，手机随时可用，
数据不出本地，底座能力持续增长。


### 长期：自主进化的个人智能体

底座足够强壮之后，系统从"工具"变成"助手"。

知识管理：
- 知识库存储可能迁移到类 Obsidian 的开源知识管理工具
  （markdown + YAML frontmatter，支持人机共编）。
  具体选型（Logseq / Foam / Dendron / 自建）由中期阶段的实际使用数据决定，
  不提前绑定产品。
- 知识不只是被动积累，系统能主动发现知识间的矛盾、过时、重复，
  自动触发审查和清理（kb_hygiene + event-trigger 已接通）。

编码能力：
- Claude CLI 的核心编码模式经过 L2/L3 吸收后，底座具备独立的代码审计、
  安全检查、补丁应用能力，不完全依赖外部编码工具。

吸收生态：
- 右花协议成为一个通用接口，任何新的开源项目都可以快速接入做评估。
- 吸收治理框架经过多次实践迭代，证据等级和吸收等级的判断标准越来越准确。
- 底座通过持续吸收，能力边界不断扩展，但复杂度不失控——
  因为每次吸收都经过核心-外围切割，只内化核心，外围配置化。

## 技术债务待办（留给 v3.0 大规模重构）

当前刻意不动、但未来大规模重构时应统一处理的技术债务，单列于此，避免散落各处被遗忘。

- **待办：未来 v3.0 大规模重构时，统一执行全量 gofmt 清理并添加强制门禁。**
  当前 `gofmt -l .` 约 115 文件"漂移"，根源是项目手写的 `/* */` 散文块注释与 Go 1.26.1
  gofmt 的规范化（正文移出 `/*` 行 + 续行 tab 缩进）+ struct tag 重排冲突。**现在不动**的理由：
  重排会降低文档注释可读性、触碰冻结的 `kernel/`、制造淹没 `git blame` 的巨型 churn
  （详见 `MERGE_DECISIONS.md` 决策 14 与 `DESIGN_PRINCIPLES.md`「代码格式立场」）。
  **为何留到 v3.0**：届时本就要大面积改代码、git 历史要翻一遍，格式 churn 的边际成本最低。
  执行项：一次全量 `gofmt -w` → 把 `gofmt -l` 检查加进 CI gate（`core_gate.sh` /
  `integration_check.sh`）→ 之后新代码强制 gofmt-clean。**前提**：若 v3.0 仍保留 kernel 冻结，
  需先回答"是否接受为格式统一而修改 `kernel/`"（按内核冻结治理的特许令机制评估）。

## 三角支撑 + 生长轴

三个要素相互增强：

个人知识库 — 越用越聪明的核心。对话中的经验沉淀下来，下次自动检索出来。
通用工具能力 — 搜索、文件操作、终端、工作流，日常工作助手的基础能力。
本地私有部署 — 数据主权。远程控制的前提是信任，信任的前提是存储不出本地。

贯穿三个要素的生长轴：
右花吸收机制 — 让底座持续变强的引擎。不是一次性建设，是持续进化。

---

## 核心竞争力

和通用 LLM（Claude、ChatGPT）的区别：
它们每次对话从零开始。beishan-core 记得你的历史，在正确的时候递给你正确的经验。

和其他 Agent 框架的区别：
它们追求能力广度，依赖外部工具生态。
beishan-core 通过吸收机制把外部能力内化，对外部项目的依赖会随吸收而减弱，
底座越来越强——不是"不依赖外部"，而是"依赖是暂时的，吸收是方向"。

和静态知识库的区别：
它们只存不长。beishan-core 的知识库有校准、有审查、有自动入库，
底座通过吸收不断获得新能力来更好地管理和使用知识。
