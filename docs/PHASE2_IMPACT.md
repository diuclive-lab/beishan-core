# 第二阶段方案项目影响推演

> 评估 TwinFlower 融合第二阶段对 beishan-core 项目在架构、性能、维护、用户、开发、开源、风险 7 个维度的影响。

---

## 1. 架构影响

| 模块 | 改动范围 | 影响层级 | 风险 |
|------|---------|---------|------|
| 澄清契约 | 新增 internal/clarify/ | internal/ | 无 |
| bench 评估 | 新增 internal/bench/ | internal/ | 无 |
| evidence | 新增 internal/observatory/evidence.go | internal/ | 无 |
| 茎注册表 | 新增 internal/registry/ + main.go +1行 | internal/ + cmd/ | 低 |
| 3 个工具 | 新增 internal/tools/{weather,translate,currency}.go | internal/tools/ | 无 |

**关键发现**：
- `kernel/`（3 个文件）**零改动**，冻结不变
- `glue/` **零改动**
- `plugins/` **零改动**（已在 Phase 1 确认不当迁移 L4 编排器）
- `workflows/`（31 个 YAML）**零改动**
- main.go 仅增加 1 行 `registry.Lock()`
- 所有新增代码在 `internal/` 内，受 Go 编译器强制隔离

**架构影响评级**：🟢 极低

---

## 2. 性能影响

| 模块 | 运行时开销 | 说明 |
|------|-----------|------|
| clarify 契约 | 单次 chat 响应增加 ~0.1ms | 纯内存 JSON 序列化 |
| bench 评估 | **零运行时开销** | 仅显式调用时执行 |
| evidence | **零运行时开销** | 纯数据结构，按需填充 |
| 茎注册表 Filter | O(96) 每次过滤 | 96 个工具遍历，< 0.01ms |
| 茎注册表 Lock | **一次启动开销** | 单次赋值 |
| 3 个工具 | **零空闲开销** | HTTP 调用时才有成本 |

**性能影响评级**：🟢 可忽略

---

## 3. 维护负担

| 模块 | 行数 | 预期变更频率 | 维护成本 |
|------|------|-------------|---------|
| 澄清契约 | 67 | 几乎不变 | 极低 |
| bench 评估框架 | 320 | 极少 | 低 |
| bench suites | 160 | 随新插件增加 | 中（每次加插件需加测试） |
| evidence | 210 | 几乎不变 | 极低 |
| 茎注册表 | 389 | 几乎不变 | 极低 |
| weather 工具 | ~100 | 外部 API 变更时 | 低（Open-Meteo 稳定） |
| translate 工具 | ~100 | 外部 API 变更时 | 低（LibreTranslate 稳定） |
| currency 工具 | ~130 | 外部 API 变更时 | 低（open.er-api.com 稳定） |

**高维护项**：bench suites（160 行）。每新增一个插件或工具，按规范应补充对应的 eval suite。这不是"额外负担"——属于最佳实践。

**低维护项**：其余 ~1,100 行几乎不变化。

**维护影响评级**：🟢 低（仅 suites 需要持续关注）

---

## 4. 用户/API 影响

**当前状态**：clarify 在 YAML 工作流中**没有任何消费者**。实际验证：

```bash
grep -rn "clarify" workflows/*.yaml
# → 无输出
```

这意味着 clarify 的返回格式从纯文本切换为 JSON **不会破坏任何现有工作流**。

**影响清单**：

| 变更 | 影响范围 | 需要适配 |
|------|---------|---------|
| clarify 返回 JSON | 零（无消费者） | 无 |
| 新增 weather 工具 | 正向（新能力） | 无 |
| 新增 translate 工具 | 正向（新能力） | 无 |
| 新增 currency 工具 | 正向（新能力） | 无 |
| registry.Lock() | 无（内部改动） | 无 |

**用户影响评级**：🟢 无破坏性变更，增加 3 个新能力

---

## 5. 开发工作流影响

| 变更 | 对开发者的影响 |
|------|---------------|
| 新增 internal/registry/ | 注册新工具时需在 `tools.Init()` 之前完成注册。与现有流程完全一致 |
| 新增 internal/bench/ | 为插件写 eval suite 成为可选的最佳实践。降低新插件引入的门槛 |
| 新增 clarify 类型 | 处理澄清逻辑时使用结构化类型而非字符串拼接 |
| 新增 evidence | 为 observability 扩展提供因果追踪，但不强制使用 |

**开发影响评级**：🟢 正向，降低测试门槛

---

## 6. 开源项目感知

| 新增能力 | 对开源贡献者的感知 |
|---------|------------------|
| bench 评估框架 | "这个项目有质量保障体系" — **正向** |
| 茎注册表生命周期 | "设计成熟，考虑运行时安全" — **正向** |
| 3 个免费工具 | "拿来就能用，有自己的天气/翻译工具" — **正向** |
| clarify 结构化类型 | "API设计规范" — **正向** |

**不引入的风险**：若跳过 Phase 2，开源贡献者看到的是 96 个工具但无评估框架、无注册保护。这意味着贡献者引入新工具时没有标准化流程可遵循。

**开源影响评级**：🟢 正向，提升项目专业度

---

## 7. 风险：不做的代价

| 不做的模块 | 代价 |
|-----------|------|
| 跳过澄清契约 | observatory 无法结构化消费 clarify 数据。"clarify→用户选择→学习"闭环断链。EWMA 增强已做了但无结构化数据消费方 |
| 跳过 bench 评估 | 31 个 YAML 工作流 + 96 个工具无量化质量指标。回归只能靠手动烟雾测试。随着工具数增长，回归覆盖成本线性增加 |
| 跳过茎注册表 | 目前问题不大。但在第二阶段后处理 66（FangLab）融合时，66 有 2 套自建注册表，无生命周期门控会增加合并冲突 |
| 跳过 3 个工具 | weather/translate/currency 继续是能力缺口。外部贡献者若添加这些工具可能采用不同 API 风格，增加不一致性 |

**最关键的不做风险**：跳过 bench 评估框架的代价最高。随着项目增长（96 tools → 更多），缺少量化评估会使回归检测越来越困难。

---

## 总结

| 维度 | 评级 | 关键结论 |
|------|------|---------|
| 架构 | 🟢 极低 | 所有新增代码在 internal/，kernel/glue/plugins 零改动 |
| 性能 | 🟢 可忽略 | 零运行时开销，仅 registry.Lock() 为一次启动操作 |
| 维护 | 🟢 低 | ~1,100 行几乎不变化，仅 suites(160行)需持续补充 |
| 用户 | 🟢 无破坏 | clarify 无工作流消费者，3 个新工具全为正向 |
| 开发 | 🟢 正向 | bench 降低测试门槛，clarify 类型提升代码质量 |
| 开源 | 🟢 正向 | 评估框架+生命周期门控提升项目专业度 |
| 不做风险 | 🟡 中等 | bench 缺失随着工具数增长回归成本线性增加 |

**结论**：第二阶段方案对项目无负面冲击。所有新增能力是纯增量，零破坏性变更。跳过 bench 是中风险——第一阶段已经通过 observatory 提升了可观测性，但缺少量化评估的能力。宜在 Phase 2 补齐。
