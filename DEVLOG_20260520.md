# 开发日志 2026-05-20

## 项目定位

经过多轮讨论，最终确定项目定位：

**Deterministic Personal Knowledge Runtime**

不是：
- AI Memory Framework
- Agent Cognitive Architecture
- 通用 agent memory 系统

而是：
- Personal Knowledge OS with Hardened Retrieval
- 确定性 > 潜在智能度
- 真实需求 > 架构完美性

---

## 核心设计原则

| 原则 | 说明 |
|------|------|
| 硬化层优先 | 能用代码写的逻辑绝不写在提示词里 |
| 工具 > Agent | 检索是写死在代码里的，不是 LLM 决定的 |
| LLM 只负责生成 | LLM 被压到 terminal renderer 角色 |
| 确定性检索 | SearchMemory 是确定性代码，不是 LLM 决策 |
| 有界注入 | 只在 think_plugin 注入，不污染其他插件 |
| 可审计 | 检索过程可追踪，写入需用户确认 |

## 方案探索过程（5 轮推演）

### 第 1 轮：硬化检索 + 自动注入（废弃）

**方案**：每次聊天自动调用 KnowledgeRetrieve（BOW + 类型权重），结果注入 system prompt。

**问题**：
- L4（think_plugin）直接 import L3（internal/tools），突破架构边界
- 将判断权交给了 LLM，违反硬化层原则
- 知识注入后无法验证 LLM 是否真的使用

**结论：废弃。**

### 第 2 轮：全局 Context Enrichment Layer（否定）

**方案**：在 L1 Router 之后、L4 Plugin 之前，加一个全局富化层，所有消息都自动检索知识。

**推演问题**：
- workflow_plugin、terminal_plugin、todo_plugin 完全不需要知识
- "帮我跑一下开源雷达" → 检索到 agent 设计模式 → 干扰执行
- "你好"、"现在几点" 也要多一次 knowledge_search → 增加延迟

**结论：否定。** memory 应该按 runtime role 注入，不是全局 middleware。

### 第 3 轮：图结构神经网络（否定）

**方案**：L3 建知识图谱（带类型的关系边），L4 消费。

**推演问题**：
- L4 不会主动消费这个神经网络
- L3 建了一张死图，没有智能度
- 本地知识库并不比 API 优秀

**结论：否定。** 核心认知：本地知识库的优势从来不是"智能"，而是精确的、个人的、确定性的事实。

### 第 4 轮：插件级自动注入（采纳）

**方案**：只在 think_plugin 的 OnMessage 中，在调 LLM 之前固定执行一次知识检索，结果注入 `<background>` 标签。

```
think_plugin.OnMessage:
  用户消息
    → 1. SearchMemory(query)     ← 固定步骤，不是 LLM 决策
    → 2. FormatForPrompt(top-3) ← 确定性格式化
    → 3. callLLM(msg + background) ← LLM 只做生成
```

**与原则对账：**

| 原则 | 是否违反 |
|---|---|
| 硬化层原则 | ✅ 检索是确定性 L3 工具调用 |
| 工具 > Agent | ✅ 检索是写死在代码里的 |
| 提示词只描述格式 | ✅ system prompt 只多了知识内容 |
| 内核冻结 | ✅ 不动 L1 |
| 厂商无关 | ✅ embedding 通过环境变量配置 |
| 避免递归 | ✅ 用 llm.ChatCompletion，不走 think_plugin 路径 |

**结论：采纳，开始实现。**

### 第 5 轮：双链知识图谱（采纳）

在检索上线后，进一步发现：即使 Agent 会自动检索了，每条知识仍是孤立的。

**方案**：激活 KnowledgeEntry 已有的 Links 字段，实现：

1. **入库自动建链**：新条目入库后异步扫描已有条目，基于标签/主题/关键词重叠自动双向写入 Links
2. **图遍历检索**：SearchMemory 在直接命中后，沿 Links 做一跳扩展，二跳条目降权 50% 后一起注入

**结论：采纳。**

## 自主学习方案讨论

### 核心问题

是否增加智能体的自主学习能力？

### 讨论过程

**方案 A：自动写入（否定）**

对话结束自动判断是否值得记住。

**问题**：
- 自动写入比自动检索危险得多
- 错误知识 → 入库 → autoLink → embedding → graph expand → retrieval → 强化污染
- 形成正反馈污染环
- 违反确定性原则

**结论：否定。**

**方案 B：Suggest-to-Remember（采纳）**

Agent 检测 → 提议 → 明确确认 → knowledge_remember。

**流程**：
```
think_plugin
回答完成
↓
DetectCandidateKnowledge()
↓
candidate found
↓
"你似乎刚验证了一个稳定结论：
『M2 8GB 本地模型路线暂停』
是否加入知识库？"
↓
confirm
↓
knowledge_remember(...)
```

**优点**：
- 显式：用户看到提议，明确确认
- 有界：只在 think_plugin，不污染其他
- 可审计：用户知道什么被提议
- 确定性：触发条件是代码，不是 LLM
- 可回滚：没确认就不写

**结论：采纳。**

**方案 C：Scheduler Knowledge Review（采纳，但降级）**

定时审查对话，生成报告（不自动写入）。

**原方案**：scheduler → session_search → think_plugin analyze → report

**问题**：
- LLM pipeline nesting
- prompt drift
- extraction instability

**降级方案**：先做 deterministic review（基于关键词触发），不做 LLM 分析。

**结论：采纳，但先做 deterministic review。**

## 最终确定的三个 Stage

### Stage 1: Retrieval Trace

**目的**：可观测性，知道检索过程发生了什么。

**实现**：
- `RetrievalTrace` 结构体，context-style collector
- `SearchMemory` 添加 trace 参数
- think_plugin 打印到日志

**输出示例**：
```
[retrieval] query="本地模型方案怎么样了？" stages=2
  stage=keyword_expand method=llm_expand+score output=map[candidates:6 keywords:[本地模型方案 本地模型 local model 方案进展 调研]]
  stage=ranking method=sort+limit output=map[final:3]
```

### Stage 2: Suggest-to-Remember

**目的**：主动提议记住，用户确认后才写入。

**实现**：
- `PendingRemember` 结构体，TTL 10 分钟
- 触发关键词：已放弃、最终方案、确认结论、架构决定、经验教训、踩坑记录、最终决定
- think_plugin 检测触发词，提议记住
- 用户回复"确认"，调用 knowledge_remember

**流程**：
```
用户: "本地模型方案最终决定放弃了吗？"
Agent: "是的...已放弃..."
Agent: "是否将此结论加入知识库？回复「确认」即可。"
用户: "确认"
Agent: "已记录：最终决定"
```

### Stage 3: Scheduler Knowledge Review

**目的**：定时审查对话，提取值得记住的知识候选。

**实现**：
- 工作流 `knowledge_review_scheduler`
- 基于关键词触发（deterministic review）
- 只生成报告，不自动写入
- 人工确认后才入库

**流程**：
```
scheduler 定时触发
  ↓
扫描最近对话记录
  ↓
提取包含触发关键词的对话
  ↓
生成审查报告
  ↓
人工确认
  ↓
knowledge_remember
```

## 关键设计决策

### 决策 1：不做自动写入

**原因**：自动写入会导致正反馈污染环，违反确定性原则。

**替代方案**：Suggest-to-Remember，用户确认后才写入。

### 决策 2：不做 LLM 分类

**原因**：LLM-driven semantic ontology 会侵蚀 precision、determinism、auditability。

**替代方案**：代码化的触发关键词，确定性检测。

### 决策 3：先做 autonomous discovery，不做 autonomous mutation

**原因**：发现值得记住的内容（discovery）比自动写入（mutation）安全得多。

**路线**：Retrieval Trace → Suggest-to-Remember → Scheduler Review → 半自动确认 → 谨慎考虑自动写入。

### 决策 4：PendingRemember 使用 TTL

**原因**：防止"今天 suggestion，明天确认"污染 session 状态。

**实现**：TTL 10 分钟，过期自动清理。

### 决策 5：触发关键词保守化

**原因**：宁可少建议，不要疯狂 Suggest-to-Remember。

**实现**：只检测明确的结论标记：已放弃、最终方案、确认结论、架构决定、经验教训、踩坑记录、最终决定。

## 关键实现细节：Session-Scoped PendingRemember

### 问题发现

初始实现中 PendingRemember 是全局的（单变量），存在两个问题：

1. **跨会话碰撞**：多个用户同时使用时，后一个会话会覆盖前一个会话的 pending remember
2. **HTTP 无状态**：每次 HTTP 请求生成新的 sessionID，导致"确认"请求找不到之前的 pending

### 解决方案

**Step 1: 添加 SessionID 到 Message（L1 改动）**

```go
// kernel/msg.go
type Message struct {
    ...
    SessionID string `json:"session_id,omitempty"` // 会话 ID，用于 request-scoped 状态
}
```

**讨论**：这是 L1 kernel 改动。SessionID 是请求级元数据，与 ReplyTo、CorrelationID 同级。虽然可以放在 Payload 里，但语义上更清晰。用户最终接受此方案，但要求以后动 L1 前必须多次提醒确认。

**Step 2: HTTP handler 支持客户端传入 session_id**

```go
// main.go
sessionID := ""
if sid, ok := raw["session_id"].(string); ok && sid != "" {
    sessionID = sid
}
if sessionID == "" {
    sessionID = newSessionID()
}
```

**Step 3: 响应包装附加 session_id**

```go
wrapped := map[string]interface{}{
    "session_id": sessionID,
    "sender":     resp.Sender,
    "type":       resp.Type,
    "payload":    json.RawMessage(resp.Payload),
}
json.NewEncoder(w).Encode(wrapped)
```

**Step 4: Session-scoped PendingRemember**

```go
// plugins/think_plugin.go
var (
    pendingRemembers   = make(map[string]*PendingRemember)  // sessionID -> PendingRemember
    pendingRemembersMu sync.Mutex
)

func createPendingRemember(sessionID, title, summary string) *PendingRemember {
    pendingRemembersMu.Lock()
    pendingRemembers[sessionID] = pr // 每个 session 只保留一个 pending
    pendingRemembersMu.Unlock()
    return pr
}
```

**Step 5: Web UI 维持 session 状态**

```javascript
// web/index.html
let lastSessionId = null;

// 发送请求时携带上次的 session_id
if (lastSessionId) {
    payload.session_id = lastSessionId;
}

// 响应中存储 session_id
if (d.session_id) {
    lastSessionId = d.session_id;
}
```

## L1 改动讨论记录

### 问题

实现 Suggest-to-Remember 的 session-scoped PendingRemember 时，需要在 Message 中传递 sessionID。有两种方案：

| 方案 | 优点 | 缺点 |
|------|------|------|
| Message.SessionID（L1 改动） | 语义清晰，与其他元数据一致 | 修改了冻结的 L1 |
| Payload 内嵌 session_id | 不动 L1 | 语义别扭，混在业务数据里 |

### 决策

采用 Message.SessionID 方案，原因：
1. SessionID 是请求级元数据，与 ReplyTo、CorrelationID 同层
2. 只是 struct 增加一个 `omitempty` 可选字段，不影响内核路由逻辑
3. 其他插件也可以使用（如 future session-scoped 状态）

### 新增规则

**以后修改 L1 kernel 代码前必须多次提醒用户确认，不能直接修改。**

## 核心认知

1. **本地知识库的优势不是"智能"，而是精确的、个人的、确定性的事实。**

2. **自动写入比自动检索危险得多。** 错误知识会形成正反馈污染环。

3. **先做 autonomous discovery，不要立刻做 autonomous mutation。**

4. **vision 的价值是帮助拒绝错误方向，不是提前实现全部未来能力。**

5. **新抽象必须回答一个已发生的 bug。** 系统已经工作，不要进入"架构兴奋期"。

6. **修改 L1 前必须多次确认。** L1 是微内核，设计上应该冻结。
