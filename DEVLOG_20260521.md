## 三个缺口收尾（编号 H1-H3，区别于主线 S 系列）

### H1 — content_review.yaml on_error 加固

**为什么做：** `cross_check`（web_search）和 `kb_check`（knowledge_search）是网络依赖步骤，超时或 API 不可用会终止整个工作流。加 `on_error: evaluate` 让网络失败不阻塞，评估步骤仍可基于已有数据出结果。

**改动：** `workflows/content_review.yaml` 两个步骤各加一行 `on_error: evaluate`。`internal/workflow/types.go` 的 `StepDef` 已有 `OnError` 字段且 engine 已支持，只需改 YAML。

### H2 — SessionSearchStructured maxAge 过滤

**为什么做：** 会话检索只增长不淘汰，过期对话污染检索结果，且增加 IO 开销（每搜索扫描全量会话文件）。

**改动：** `tools.SessionSearchStructured(keyword, limit)` → `(keyword, limit, maxAgeDays)`，默认 30 天。在检索循环中跳过 `ModTime` 早于 cutoff 的文件。调用方 `retrieval_pipe.go:233` 传 30。不删除文件，只做检索时过滤。

**实测：** 冒烟测试 12/12 通过，会话检索正常。

### H3 — retrieval trace 注入响应 payload

**为什么做：** 后端 `RetrievalTrace` 数据已存在（`knowledge.go:58-122`），但只走 `fmt.Printf` 到日志。前端无法拿到结构化 trace 展示。

**改动：** 新增 `ModeTrace`（`"trace"`）模式，`think_plugin.go` 的 `handleChat` 在 `wantTrace=true` 时返回 `{"reply": "...", "retrieval_trace": {...}}` 结构化 JSON。暂不改前端，只保证数据能传出。

**决策理由：** 不默认启用（避免破坏已有 JSON 响应格式），通过 `mode: trace` 显式开启。workflow 步骤用 `no_retrieval` 无检索，所以无冲突。

## 三个 M 类任务

### M5 — 知识过期/归档机制

**为什么做：** 知识库只增不减是定时炸弹。现有 `Ephemeral` + `ExpiresAt` 只支持逐条过期，缺少批量状态管理。

**做法：**
1. `KnowledgeEntry` 新增 `Status string`（`active`/`archived`/`expired`，空=active）和 `LastAccessedAt int64`
2. `SearchWithScore` 和 `searchByEmbedding` 的循环开头过滤 `status != "" && status != "active"` → 直接 `continue`
3. `recordAccess(id)` 函数在检索命中或引用时写入 `LastAccessedAt`，调用点覆盖 `SearchWithScore` 的匹配结果和 `loadLinkedEntries` 的已加载条目
4. `kb_audit` 检测 `LastAccessedAt > 0 && LastAccessedAt < now - 90days && status != "archived"` 的条目，标记为 `severity: "info"` 的审计问题

**效果：** 用户通过 `kb_audit` 可以看到「90天未访问，建议归档」的建议，手动调 `knowledge_update` 设 `status: "archived"` 后条目自动从检索结果中排除。不自动归档，由用户决策。

### M2 — 知识图谱可视化

**为什么做：** `KnowledgeTopicMap()` 只输出文本标签聚类，不是图结构。TypedLinks 存储了条目的有类型关联（related/contradicts/supersedes/supports），但没有可视化方式展示。

**做法：**
1. 后端 `KnowledgeGraph()` 工具读取全量条目，输出 `{"nodes": [{id, title, source_type, tags}], "edges": [{source, target, relation}]}` JSON
2. 前端 `web/index.html` 侧栏加「📈 知识图谱」入口，点击打开模态框
3. D3.js（从 cdnjs 7.9.0）渲染 force-directed graph，节点颜色按 `source_type` 区分
4. 支持拖拽、缩放、点击节点自动填充 ID 到输入框

**实测：** `knowledge_graph` 返回正确数据结构（28 nodes, 0 edges — 当前库无 typed_links 所以无边，结构本身完整）。

### M3 — L2 glue 激活（架构验证）

**为什么做：** glue 层代码（`glue/glue.go` ~260 行）有完整的 IPC 协议实现（stdin/stdout JSON dispatch、子进程生命周期管理、5s register 超时），但之前无真实的 Python 业务插件运行，属于「框架就绪，业务缺失」状态。需要验证四层架构（L1 内核 → L2 glue → Python 子进程）在实际场景下可用。

**做法：**
1. `plugins/web_render_py/manifest.json` — 声明插件名和入口
2. `plugins/web_render_py/main.py` — 完整 IPC 协议实现：
   - 启动后发 `{"type": "register", "name": "web_render_py"}`
   - 主循环读 stdin，处理 dispatch（支持 web_render 消息类型，用 playwright-python 渲染页面）
   - 收到 shutdown 消息退出
3. `plugins/web_render_py/requirements.txt` — playwright 依赖

**实测效果：**
```

[web_render_py] registered, waiting for dispatch...
2026/05/21 16:42:12 [Kernel] 插件注册: web_render_py
```

Glue 层自动完成了：
1. 扫描 `plugins/` 发现 `web_render_py/manifest.json`
2. 检测 `requirements.txt` 自动 `pip3 install -r requirements.txt`
3. `exec.Command("python3", "main.py")` 启动子进程
4. 等待 5s 内收到 register 确认
5. 注册到内核，web_render_py 成为可路由插件
6. 静态检查：`grep -rn 'tools\.Execute(' plugins/` 通过（Go 工具不受影响）

**架构验证结论：** L2 glue IPC 协议在真实业务场景下可用，子进程启动耗时约 1s（含 pip install），消息往返延迟微秒级。四层架构可扩展性得到验证。

## 核心认知

1. **kernel.Call 到同一插件会死锁** — `k.Send` → `plugin.OnMessage` 是同步调用，re-entrant 不返回。同一插件的内部调用必须用方法调用，不能走内核路由。
2. **sync.Once 是 Go 标准惰性初始化模式** — 浏览器单例 3 个包级变量 + 15 行 init 函数，改动风险极低，收益是每次调用省 3-8s 冷启动。
3. **Glue IPC 协议已验证可用** — 子进程 stdin/stdout JSON dispatch 延迟微秒级，5s register 超时合理。四层架构（L1 路由 → L2 IPC → Python 子进程）可扩展。
4. **入口门禁比出口清理更有效** — KnowledgeEntry.Status 在检索时过滤比定期清理更轻量。检索函数加 2 行 `continue` 比 Cron 扫库更可靠。

## 晚上：第三轮测试 + 工具链质量修复

### 第三轮测试暴露的系统性问题

做了三轮真实 API 测试，前两轮覆盖基础对话和记忆连续性（6 个场景全部通过），第三轮工具链测试 3 个全失败：

| 测试 | 期望 | 实际 |
|------|------|------|
| T7 搜索 Go 热门项目 | 返回总结结果 | 返回了无关的备案信息，无 LLM 总结 |
| T8 列出待办 | 返回待办列表 | `todo_plugin` 报"未知类型 list" |
| T9 内容入库 | 入库成功 | `source_type` 缺失被 schema 拦截 |

三个失败指向同一根因：**路由层（DeepSeek）的工具调用质量不足**。LLM 生成的 type 名和参数不精确。

### 根因分析

两个子问题：

**子问题 1：DeepSeek 不知道精确的 type 名**
`todo_plugin` 传了 `list` 而不是 `todo_list`，`knowledge_add` 漏了 `source_type`。原因是 Router prompt 只列出了插件描述和 type 列表，但 DeepSeek 需要看示例才知道每个 type 需要什么参数。

**子问题 2：搜索结果没有回传 LLM 做总结**
```
修前链路：用户 → search_plugin → 原始 JSON → 用户
修后链路：用户 → search_plugin → think_plugin(总结) → 自然语言 → 用户
```

### 方向A：Router prompt 加示例调用

**为什么这样做：** Description + Types 提供了"是什么"但没提供"怎么用"。DeepSeek 需要看到示例调用才能理解 `todo_list` 和 `list` 的区别。

**做法：**
1. `kernel.Meta` 新增 `Example string` 字段
2. `kernel/router.go` 的 `buildPluginList()` 在渲染插件列表时追加 `示例: type="todo_list"` 行
3. `main.go` 中 7 个关键插件加了示例（search/todo/memory/terminal/browser/write/think）

**效果：** Router prompt 中每个插件的描述从 1 行扩展到 2-3 行，DeepSeek 在做路由决策时能看到精确的 type 名和参数示例。这条改动在 prompt 层，不影响任何运行时行为。

### 方向B：search_plugin 结果回传 think_plugin 汇总

**为什么这样做：** 搜索插件返回原始 JSON 结果给用户，体验差且信息密度低。需要 LLM 把搜索结果组织成自然语言回答。

**做法：**
1. `SearchPlugin` 增加 `Kernel *kernel.Kernel` 引用
2. `web_search` 处理逻辑中，执行搜索后调 `p.Kernel.Call(think_plugin, chat, no_retrieval)` 做总结
3. 总结失败时降级返回原始结果
4. `main.go` 注册时传 `SearchPlugin{Kernel: k}`

**效果：** 搜索请求现在返回 `type: chat.response` + 自然语言总结，而不是原始 JSON。

### 附带发现的 kernel 运行时 bug

在实现方向 B 时，发现 search_plugin 的代理调用（search_plugin → think_plugin → 返回）触发了一个 kernel 的隐藏 bug：

**Bug 1：`response.Sender == ""` 拦截 proxied 响应**

```go
// 修前 — 当 search_plugin 返回 think_plugin 的响应时，
// Sender="think_plugin" != ""，deliverResponse 不执行，
// Call 的 pending channel 永远收不到响应，120s 超时
if err == nil && msg.CorrelationID != "" && response.Sender == "" {

// 修后 — 只要有 CorrelationID 就回传，Sender 字段会被覆盖
if err == nil && msg.CorrelationID != "" {
```

**Bug 2：`deliverReply(msg)` 回传路由元数据而非实际响应**

```go
// 修前 — 回传的 payload 是 DeepSeek 的路由决策结果
k.deliverReply(msg)

// 修后 — 回传的 payload 是插件的实际响应
k.deliverReply(response)
```

两个 bug 都是在 proxy 模式下才暴露的（一个插件代理调用另一个插件）。在直调模式下（search_plugin 直接返回），Sender 为空所以 Bug 1 不触发，msg.Payload 就是结果所以 Bug 2 不影响。这是 proxy 模式第一次真实运行才发现的边界条件。

### 事实核查系统

**为什么这样做：** 用户说"记住：知识库有283条，健康度96分"时，系统直接写入记忆，没有验证这个事实是否真实。违反"LLM只做生成不做决策"原则的另一面——写入记忆时应该有事实验证。

**做法：**
1. `internal/tools/fact_check.go`（新文件）— `FactCheck()` 用系统工具核查可验证事实：
   - "知识库N条" → 实际统计条目数对比
   - "健康度N分" → 调 KBAudit 实际计算对比
   - 其他无法核查的事实 → `unverified` 不拦截
2. `KnowledgeEntry` 新增 `Verified bool` / `VerifiedAt int64` 字段
3. `computeStructuralBoost` 对 `Verified=true` 的条目 +2 加权（检索时排在前面）
4. `confirmPendingRemember` 中插入 FactCheck — 矛盾时告知用户实际值，询问是否强制记录

**效果：** 错误断言被拦截（如"知识库有999条" → "实际是29条，是否强制记录"），正确断言被标记为 `verified` 获得检索加分。

## 最终核心认知

1. **Router prompt 是 LLM 工具调用的质量瓶颈** — Description + Types 只回答了"有什么"，没回答"怎么用"。一个 `Example` 字段比三行描述更有效。
2. **Proxy 模式（插件代理调用另一个插件）是架构缺口测试** — 直调路径下隐藏的边界条件（response.Sender、deliverReply）在 proxy 第一次运行时全部暴露。
3. **写入时事实核查是"LLM 不做决策"的另一面** — 前一轮硬化层修了"L3 校验阻止 LLM 传错误参数"，这一轮是"写入前验证阻止 LLM 记错误事实"。
4. **session_id 不应污染业务 payload** — 从 payload 移到 kernel.Message 字段后，所有 schema 校验的 session_id 问题自动消失。

### S3: 批量内容导入管道

**为什么做**：content_review 是单 URL 审核，导入 10 篇文章需手动发 10 次请求。

**方案**：workflow engine 新增 `batch` 循环执行能力。

**改动**：
- `types.go`: StepDef 新增 `Batch *BatchDef` 字段，BatchDef 含 `foreach` 表达式
- `engine.go`: `runBatch()` 方法 — 求值 foreach 为数组，逐个执行步骤的 plugin:type，结果存入 ctx
- `batch_ingest.yaml`: 新工作流 — URL 列表 → content_review 循环 → 汇总报告

**实测**：2 个 URL 批量导入，`batch review [1/2]` + `[2/2]` 依次完成，汇总报告正确生成。

**附带修复**：kernel.go `deliverReply` 收到的是 response（空 ReplyTo），不是原始 msg。修复：`response.ReplyTo = msg.ReplyTo`。

### 核心认知

1. **批量循环应复用单步执行逻辑** — `runBatch` 不需要加载子工作流，直接对同一 plugin:type 循环调用，结果用 JSON 数组聚合。比 `workflow_run_batch` 轻量得多。
2. **ReplyTo 必须从原始消息传递到响应** — `deliverReply` 检查的是传入消息的 ReplyTo，如果传的是 plugin 的 response（空 ReplyTo），回程路由就断了。
3. **模板匹配比从零生成更可靠** — LLM 分类选模板比 LLM 生成完整 YAML 出错率低得多。模板保证了 no_retrieval、timeout、on_error 等最佳实践。
4. **会话摘要不需要 LLM** — 首条用户消息 + 主题词提取（确定性）足以支撑搜索。LLM 摘要是后续优化，不阻塞。
5. **memory_plugin session 工具不返回结果是预存 bug** — 所有 session 工具（search/list/cleanup）都受影响，本次一并修复。
