## 短期方向落地

### #1：并行批处理

**为什么做：** `batch_ingest` 遍历 URL 列表时串行执行，N 个 URL 需要 N × 300s。批量导入 5 篇文章需要 25 分钟。

**做法：** `BatchDef` 新增 `Parallel bool` + `Concurrency int`。`runBatch` 用 goroutine + buffered channel semaphore 实现并发控制：

- 串行时直接调用（行为不变）
- 并行时每个元素一个 goroutine，semaphore 限流 concurrency（默认 5）
- `ctx["item"]` 用 save/restore 模式保护，每个 goroutine 在开始时保存、结束时恢复父上下文
- 结果按索引写入预分配 slice，失败通过 mutex 追加
- 过滤空 output（并发下失败的位置为空字符串）

**batch_ingest.yaml** 启用 `parallel: true, concurrency: 3`。

### #2：瀑布流穿透

**为什么做：** 检索瀑布流只显示输出摘要（120 字），用户无法展开查看检索阶段的完整数据。

**做法：** `renderTraceWaterfall` 每阶段卡片加 `onclick` → `toggleTraceStage(id)`。长内容默认折叠，点击展开完整 JSON（含 entry ID、分数等）。短内容直接显示，无折叠。

## 三个紧急修复

### 修复 1：skill_factory RecordUsage

**问题：** `classifyTemplate` 和 `generateWorkflow` 都调了 `llm.ChatCompletion`，但都没有 `RecordUsage`。skill_factory 生成的 YAML 工作流消耗的 token 完全不可追踪。

**修法：** 两处改为 `llm.ChatCompletionWithUsage` + `RecordUsage("skill_factory", usage)`。模式和 think_plugin 完全一致。

### 修复 2：多轮对话 token 截断

**问题：** `loadRecentSessionMessages(sessionID, 5)` 加载历史后直接 append 到 messages。如果 5 轮对话都很长（含大代码块），token 总量可能超过模型窗口。

**修法：** 从最新往前累计 `len([]rune(content))`，超 8000（≈6K token，留余量）时截断最早的历史。保证最近的上下文一定在。

```go
const maxHistoryRunes = 8000
total := 0
var safe []llm.ChatMessage
for i := len(history) - 1; i >= 0; i-- {
    size := len([]rune(history[i].Content))
    if total+size > maxHistoryRunes {
        break
    }
    total += size
    safe = append([]llm.ChatMessage{history[i]}, safe...)
}
history = safe
```

### 修复 3：preRoute 搜索歧义

**问题：** 用户说"搜索一下知识库里怎么说的" → 先匹配 `"搜索一下"` → 路由到 `search_plugin/web_search`（网络搜索），应该走 `knowledge_search`。原因是 `"搜索一下"` 的关键词位置在 `"搜索知识库"` 之后，但 `"搜索一下"` 也包含 `"搜索"` 字。

不对，实际问题是：`"搜索一下知识库"` 包含 `"搜索一下"`（第 35 行匹配），不包含 `"搜索知识库"`（缺少"库"字之前的部分）。两个关键词都不完全匹配中间态。

**修法：** 命中 `search_plugin/web_search` 后加二次检查：如果原始文本包含 `"知识库"` 或 `"记忆"`，改路由到 `memory_plugin/knowledge_search`。

```go
if p.recipient == "search_plugin" && p.msgType == "web_search" {
    if strings.Contains(userText, "知识库") || strings.Contains(userText, "记忆") {
        msg.Recipient = "memory_plugin"
        msg.Type = "knowledge_search"
    }
}
```

## L1 主动观察者 Agent

### 为什么做

知识库现在是"被动接收"——用户主动入库才能增长。L1 Agent 让它变成"主动观察"：每天早上自动扫一遍技术资讯源，发现与项目关注领域相关的内容，自动走 content_review 入库，最终推送给用户。

这是从「数据库」到「有感知能力的系统」的第一步。

### 架构

```
agent_observer 工作流（工作日 08:00）
  fetch（rss_default → 5 个技术源）
    → score（LLM 按关注领域筛选：Go/微内核/知识管理/量化/视觉AI）
    → auto_ingest（高相关内容触发 content_review 入库）
    → notify（汇总报告）
```

全部复用已有工具和引擎能力，新代码只有 RSS 解析（~150 行）。

### rss_fetch 工具

纯 Go 标准库实现，同时支持 RSS 2.0 和 Atom：

- XML 结构映射到 `rssFeed` / `atomFeed` 两个内联结构体
- `fetchRSS` 先试 RSS 2.0 解析，失败退到 Atom
- 统一输出 `FeedItem{title, link, description, published, source}`
- `rss_default` 预置 5 个源：HN / Go Blog / GitHub Trending Go / 掘金 / InfoQ

### 关注领域

写入 workflow 的 LLM prompt 中，无需代码改动即可更新：

```
- Go 语言开发、微内核架构、智能体编排
- 知识管理、语义检索、知识图谱
- 量化交易、股票分析、金融数据
- 图片生成、视觉 AI、提示词工程
- 开源项目工程实践、架构设计
```

### 实测

首次运行从 Hacker News 和 Go Blog 拉取 6 条：

| 来源 | 标题 | 相关性 |
|------|------|--------|
| HN | The IBM-ification of Google | 中（架构/工程） |
| HN | Memory shortage repricing consumer electronics | 低 |
| HN | Uv package management UX | 低 |
| Go Blog | pkg.go.dev API | 高（Go） |
| Go Blog | Type Construction and Cycle Detection | 高（Go） |
| Go Blog | Inliner and source-level inliner | 高（Go） |

GitHub Trending Go、掘金、InfoQ 因反爬限制失败。后续可以换成更稳定的 RSS 源。

### 核心认知

1. **RSS 解析不需要第三方库** — Go 的 `encoding/xml` 配合两个内联结构体即可处理 RSS 2.0 和 Atom。150 行替代了一个完整的 go module 依赖。
2. **LLM 筛选比关键词匹配更灵活** — 工作流的 score 步骤用 LLM 评估相关性而非硬编码关键词。关注领域写在 prompt 里，用户随时可以改。
3. **预置源必然有部分失败** — 5 个源中 3 个因反爬失败，这不是 bug。RSS 源本身是半衰期很短的资源，定期更新源列表是运维任务。

## #6 FactCheck 交互纠错闭环

### 为什么做

FactCheck 后端上周五已实现（检测可核查事实、与实际值对比、矛盾时拦截），但拦截后只返回一行文本：

```
⚠️ 事实核查不通过：知识库实际有29条，不是999条
实际值：29
仍需记录？回复「是的，强制记录」
```

用户需要手动打字回复。体验上是"被拒绝了"而不是"可以纠正"。需要前端把这条文本解析为结构化对比卡片，加一键操作按钮。

### 做法

**后端：isForceSaveReply**

新增 `isForceSaveReply(sessionID, text)` 函数，检测以下回复：

| 触发词 | 行为 |
|--------|------|
| "强制记录" | 跳过 FactCheck，直接 KnowledgeRemember |
| "是的，强制记录" | 同上 |
| "强制入库" | 同上 |

在 `confirmPendingRemember` 分歧处理之后、自然语言触发器之前插入。确保用户在收到矛盾提示后回复这些触发词时，不走 LLM 路由，直接执行强制入库。

**前端：addMsg 对比卡片检测**

修改 `addMsg` 函数，当系统消息内容包含 `⚠️ 事实核查` 时，不渲染纯文本，而是生成结构化对比卡片：

```
┌──────────────────────────────────┐
│ ⚠️ 事实核查不通过                │
│                                  │
│  你的数据   999条  (红色删除线)   │
│  实际值     29条   (绿色高亮)    │
│                                  │
│  [ 强制入库 ]                    │
└──────────────────────────────────┘
```

"强制入库"按钮调用 `forceSave()` → 填入"强制记录"到输入框 → 自动发送。

### 效果

```
修前：⚠️ 事实核查不通过 → 用户手动打字"是的，强制记录"
修后：⚠️ 事实核查不通过 → 前端弹对比卡片 → 点按钮完成
```

### 核心认知

1. **文本交互到图形交互的转换点在于前端模式匹配** — 后端不必改响应格式，前端在 `addMsg` 中检测 `⚠️ 事实核查` 模式串，匹配则渲染组件。后端保持纯文本兼容所有客户端（CLI/API/Web），前端做增强。
2. **强制记录的触发词放在后缀匹配** — "强制记录" 放在 `isForceSaveReply`，并且放在 `isConfirmReply` 之后执行。用户先说"确认"触发 FactCheck → 看到矛盾 → 再说"强制记录"触发强制保存。两个步骤自然衔接。

## L7 用户画像注入

### 为什么做

think_plugin 的 system prompt 是固定的：

```
你是 beishan-core 智能助手。你所在的系统提供以下能力：
- 搜索网络（search_plugin）
- 读写文件（write_plugin）
...
```

每个用户、每个场景收到同样的回答风格。但实际需求不同：

| 用户类型 | 期望的回答风格 |
|---------|--------------|
| 高级开发者 | 直接给代码，少解释 |
| 刚入门 | 详细解释，附带背景 |
| 量化方向 | 侧重数据和算法分析 |
| 管理岗 | 概括性结论，不纠结技术细节 |

**目标：** 根据用户画像动态调整 system prompt，让 LLM 的回答风格、技术深度、关注重点更贴合用户。

### 做法

**profile.go（新文件，~150 行）**

`UserProfile` 结构体：

| 字段 | 可选值 | 默认 |
|------|--------|------|
| `name` | 任意字符串 | 空 |
| `preferred_lang` | zh / en | zh |
| `expertise` | beginner / intermediate / expert | intermediate |
| `response_style` | concise / detailed / code_first | concise |
| `interests` | 字符串数组 | 空 |

持久化到 `~/.hermes/user_profile.json`。

**ProfileToPrompt()** 把画像格式化为一行 system prompt 后缀：

```
【用户画像】技术水平: 高级 | 关注领域: Go, 架构, 量化交易, 知识管理 | 回答偏好: 优先给代码实现
当前时间: 2026-05-22 10:30
```

**think_plugin.go** — 在构建 messages 时追加：

```go
profilePrompt := tools.ProfileToPrompt()
sysContent := systemPrompt
if profilePrompt != "" {
    sysContent += profilePrompt
}
messages = []llm.ChatMessage{{Role: "system", Content: sysContent}}
```

### 实测

设置画像：`expertise=expert, response_style=code_first, interests=[Go 架构 量化交易 知识管理]`

提问"写一个 Go 函数计算斐波那契数列"：

```
// Fib 返回斐波那契数列的第 n 项（n 从 0 开始）
func Fib(n int) int {
    if n < 2 { return n }
    a, b := 0, 1
    for i := 2; i <= n; i++ { a, b = b, a+b }
    return b
}
```

直接给了代码实现（迭代+递归+大数三个版本），然后才附说明。`code_first` 风格正确命中。

### 核心认知

1. **用户画像是 system prompt 的扩展，不是替代** — ProfileToPrompt 返回的是一段结构化描述，追加在现有 systemPrompt 后面而不是替换它。systemPrompt 中的能力列表、角色定义保持不变。
2. **画像数据是用户主动设置的，不是被动分析的** — 当前没有用 LLM 或统计方法从对话历史自动提取画像。原因是自动提取容易出错（把单次询问翻译误解为永久偏好），用户主动设置更可靠。
3. **5 个字段覆盖 80% 的差异场景** — name/language/expertise/style/interests 这五个维度已经能显著改变 LLM 的回答质量和方向。更多字段（如时区、职业、常用工具链）等有需求再加。

### L4：多项目空间

**为什么做：** 知识库所有条目混在同一个空间。当用户想用 beishan-core 管理多个场景（个人/工作A/项目B）时，检索互相干扰。

**做法（最小侵入）：**
- `KnowledgeEntry` 新增 `Namespace string` 字段，空="default"
- **不改变存储路径**（`~/.hermes/memory/knowledge/` 不变）
- `SearchWithScore` / `searchByEmbedding` 加 `namespace` 参数
- `matchNamespace` 辅助函数：`ns=""` 时不过滤（向后兼容）
- `knowledge_add` 工具新增 `namespace` 参数

**设计决策：** 不改变存储路径。Namespace 只是元数据过滤，不是物理隔离。路径隔离是后续优化，当前用标签过滤足够。

### L3：知识版本控制

**为什么做：** 每次 `knowledge_update` 直接覆盖文件，之前的版本就丢了。如果用户想"回退到三天前的版本"，没有途径。

**做法：**
- `saveVersionSnapshot` — 在 `knowledge_update` 写入新版本前，`loadKnowledge` 重新加载原始条目并保存快照
- 存储路径：`~/.hermes/memory/knowledge/history/{id}/v{timestamp}.json`
- `knowledge_history` — 列出指定条目的所有历史版本（时间倒序）
- `knowledge_version_get` — 查看指定版本的完整内容

**snapshot 时机 bug 修复：** 第一版把 `saveVersionSnapshot` 放在了字段修改之后，所有历史版本记录的都是新内容。修复：`saveVersionSnapshot` 前重新 `loadKnowledge` 从磁盘加载原始数据。

### L10：知识自愈

**为什么做：** 知识库只有 `kb_audit` 做结构完整性检查（缺 source_type/tags），但没有内容层面的自愈——两条极相似的内容不会自动被发现。

**做法：**
- `KnowledgeHeal` — 用 BOW 向量两两对比全量条目的余弦相似度
  - `>=85%` → 建议合并（merge）
  - `>=60%` 且无 TypedLinks → 建议关联（link）
  - 上限 20 条，按相似度降序
- 跳过空 title 条目（硬件前缀噪音过滤）
- `knowledge_heal` 工作流 + cron 每周一 04:00

**实测（51 条目）：**

| 条目 A | 条目 B | 相似度 |
|--------|--------|:------:|
| 硬化层原则（代码优先） | findSemanticLinks修复：LLM→代码判断 | 90% |
| 手动知识库状态 | 当前知识库状态(正确) | 89% |
| 本地模型方案已放弃 | 本地模型真实环境验证发现 | 86% |

**有用建议 vs 噪音：** 前三组都是有效发现（确实应合并或关联）。后面的空 Title 条目噪音来自硬件前缀 metadata 污染，不是算法问题。

### L5：检索反馈环

**为什么做：** 检索排序现在只基于关键词匹配 + 结构化加权。如果某条知识被引用后用户觉得"不对"，没有反馈机制让这条知识降权。

**做法：**
- `KnowledgeEntry` 新增 `HitCount int64` + `UtilityScore float64`
- `recordAccess` 检索命中时 `HitCount++`（自动跟踪使用频率）
- `computeStructuralBoost` 加 UtilityScore 加权：正分+上限2，负分-上限2
- `knowledge_feedback` 工具：`up`(+1) / `down`(-1) / `reset`(0)

**效果：** 用户对一条知识说"有用"→ UtilityScore+1 → 下次检索时这条知识排名上升。说"没用"→ 排名下降。越用越准。

### L6：胶水层强化（子进程健康检查 + 自动重启 + 消息限流）

**为什么做：** 5 个子进程通过 glue IPC 运行（web_render_py 等），当前没有保护机制——子进程崩溃悄无声息，下一个请求直接挂掉。也没有消息大小保护，大 payload 可能撑爆内存。

**做法：**

**健康检查（healthCheckLoop）：**
- 每 30 秒扫描所有子进程的 `alive` 状态
- `alive` 在 stdin 写入失败、stdout 读取超时、响应解析错误时自动标记为 `false`
- 发现失效子进程 → 触发 `respawn` 自动重启

**自动重启（respawn）：**
- `proc` 结构体新增 `alive bool`、`restarts int`、`lastRest time.Time`、`manifest Manifest`
- respawn 流程：找 manifest → `Kill` 旧进程 → 重新 `spawn`（含 pip install + register)
- **熔断保护：** 同一进程 5 分钟内重启超过 3 次则熔断，不再尝试，避免 fork-bomb

**消息限流（maxMsgSize=10MB）：**
- `OnMessage` 入口检查 `len(msg.Payload) > maxMsgSize` → 直接拒绝
- stdin 写入失败 → `p.alive = false`
- stdout 读取超时 → `p.alive = false`
- 标记失效后由健康检查或下一次调用触发重启

**OnMessage 自动重启：**
- 调用时检测 `!p.alive` → 尝试 `respawn` 一次 → 重试成功则继续处理
- 避免每次调用都等 30 秒健康检查周期

**实测：** 5 个插件全部启动正常（go_example/l3_echo_go/l3_echo_python/python_example/web_render_py），胶水层健康检查在后台运行。

### knowledge_export / knowledge_import（跨机器同步）

**为什么做：** 两台电脑开发，知识库无法自动对齐。拉 git 只能同步代码，知识库里的决策记录、架构笔记不同步。

## 核心认知

1. **Namespace 作为元数据过滤而非物理隔离是正确的选择** — 不改存储路径意味着零迁移成本、零数据复制。200 行改动实现多空间隔离，对比路径隔离（需迁移+双倍存储）是更好的投入产出比。
2. **版本快照的保存时机必须在修改前** — `saveVersionSnapshot` 必须在修改字段前调用，否则保存的是新版本而非旧版本。一个看似微小的顺序错误导致整条版本链的数据都错了。
3. **BOW 向量对比可以有效发现重复和缺失关联** — 85% 阈值下找到了真实的重复条目对，60% 阈值找到了缺失关联。噪音主要来自空 title 和硬件 metadata 污染，不是算法本身的问题。
4. **检索反馈的加权幅度需要克制** — UtilityScore ±2 的上限确保搜索不会因为几次反馈就被完全扭曲。高频条目的 HitCount 自然累积权重，低频条目靠 UtilityScore 加速收敛。
5. **熔断保护比自动重启更重要** — 没有熔断的自动重启是 fork-bomb。`maxRestarts=3/5min` 确保崩溃进程不会无限复活。Go 子进程的 `Kill` 是立刻的（SIGKILL），不会留下僵尸进程。
6. **OnMessage 的惰性重启比健康检查轮询更快响应** — 健康检查 30 秒周期，但 OnMessage 在发现 `!p.alive` 时立即 respawn。健康检查兜底处理无人调用的僵尸进程。

### 5 个修复

**修复 1 — Router 重复条目（最严重）**

根因：`respawn` → `spawn` → `g.kernel.Register()` → `Router.AddKnownPlugin()` 每次都 append。

修法：`spawn` 中检查 `g.procs[m.Name]` 是否已存在，已存在则跳过 `Register`：

```go
g.mu.Lock()
_, alreadyRegistered := g.procs[m.Name]
g.procs[m.Name] = p
g.mu.Unlock()
if !alreadyRegistered {
    g.kernel.Register(m.Name, g)
}
```

**修复 2 — respawn Kill 后没 Wait**

`Kill` 是异步的（发送 SIGKILL），新旧进程的 stdout/stderr 可能混在一起。修法：Kill 后 goroutine `Wait` + 3 秒超时。

**修复 3 — KnowledgeHeal 跳过空 title**

之前 88% 的假阳性（空 title 条目 ↔ 硬化层原则）是因为硬件前缀 metadata 导致 BOW 向量相似。修法：循环开头 `if e.Title == "" { continue }`。

**修复 4 — UtilityScore 分支合并**

两段完全一样的乘 2+clamp 逻辑合并为单段：

```go
us := int(entry.UtilityScore * 2)
if us > 2 { us = 2 } else if us < -2 { us = -2 }
boost += us
```

**修复 5 — export 提示补 BOW**

导出结果 message 追加"导入后请运行 knowledge_embed_all 补 BOW 向量。"

### 技术债清理

**BOW 硬件前缀过滤：** `buildEmbedText` 中 strip Summary 开头的 `【...】`硬件信息。之前两台机器的硬件信息（`【darwin/arm64 Apple M2 8核 8GB】` vs `【darwin/arm64 Apple M3】`）导致同一条知识在不同机器上向量不同，且无关条目因相同前缀被误判为相似。

### 三、11 项改进实装

#### 1. 搜索结果相关性验证（搜索管道第一道防线）

**文件：** `plugins/search_plugin.go`（+99 行）

**问题：** search_plugin 拿到搜索结果后直接丢给 think_plugin 总结，完全没校验相关性。实测搜"中国移动股票"返回视频网站结果，零拦截。

**方案：**
- `tokenize()` — 中文按字符分词，英文按空格，统一小写
- `checkSearchRelevance(query, resultJSON)` — query 关键词与结果标题/描述做重叠检测，返回 0.0~1.0 分数
- 相关性 <30% 时，在送给 LLM 总结的 prompt 中注入警告："以下搜索结果与用户查询的相关性很低，请告知用户"

**决策：** 改为在 search_plugin 内部做检测而非 think_plugin 输出门禁，因为搜索结果相关性是搜索管道自己的问题，不应该等到输出端才拦截。

#### 2. Import 冲突警告（数据安全）

**文件：** `internal/tools/sync.go`（+53/-17 行）

**问题：** `knowledge_import` 遇到同 ID 条目直接 `skipped++`，零提示。两台机器改了同一条目，后导入的修改完全丢失。

**方案：**
- 新增 `force` 参数，`force=false`（默认）时返回 `importConflict` 详情（id/old_title/new_title/old_status）
- `force=true` 时覆盖，覆盖前调 `saveVersionSnapshot()` 自动备份旧版本到 `history/`
- 返回结构改为 `{imported, skipped, conflicts[], total, message}`

**决策：** 默认不覆盖（安全优先），但提供 `force` 选项和自动备份，让用户有选择权。

#### 3. RSS HTML 标签净化（防御性修复）

**文件：** `internal/tools/rss.go`（+21 行）

**问题：** `[:300]` rune 截断可能切碎 HTML 标签（如 `<div>...` 被切成 `<div>...<i`），前端 innerHTML 渲染时可能导致排版崩溃。

**方案：**
- `stripHTML()` — 正则 `<[^>]*>` 去标签 + 6 种 HTML 实体解码（`&amp;` `&lt;` `&gt;` `&quot;` `&#39;` `&nbsp;`）
- RSS 2.0 和 Atom 解析都在截断前先调 `stripHTML()`

#### 4. PGID 进程组强杀（安全加固）

**文件：** `glue/glue.go`（+27/-6 行）

**问题：** `cmd.Process.Kill()` 只杀直接子进程，Python 插件派生的孙进程会游离在后台造成资源泄漏。

**方案：**
- `cmd.SysProcAttr = &syscall.SysProcAttr{Setpgid: true}` — 子进程独立进程组
- `killProcessTree(cmd)` — `syscall.Kill(-pgid, SIGKILL)` 杀整棵进程树
- 替换 `spawn()`、`respawn()`、`Shutdown()` 中所有 `Process.Kill()` 调用

**决策：** 用 PGID 而非 PID namespace，因为 Darwin/Linux 都支持，无需 root 权限。

#### 5. memory_plugin 自动扫描注册（架构改进）

**文件：** `plugins/memory_plugin.go`（-35 行）、`internal/tools/tools.go`（+6 行）

**问题：** 新加工具必须在 `main.go` Meta.Types、`tools.go` Init、`memory_plugin.go` case 三处同步注册，漏任何一处就静默失效（`knowledge_graph` 就是这么漏的）。

**方案：**
- `tools.HasTool(name)` — 检查 `Registry` map 中是否存在
- `memory_plugin.OnMessage` 从 50+ 行硬编码 case 改为 `if tools.HasTool(msg.Type)` 动态转发
- 新增工具只需在 `tools.go` Init 中注册，memory_plugin 自动可达

**决策：** 不改 Meta.Types（那是给 LLM Router 看的），只改运行时转发。Meta.Types 的同步问题留到后续用代码生成解决。

#### 6. base64 噪音过滤（数据质量）

**文件：** `internal/tools/embed.go`（+19 行）

**问题：** 多模态图片入库后，`content` 中可能包含大段 base64 编码，严重污染 BOW 向量的词频分布，导致相似度计算失真。

**方案：**
- `stripBase64Noise(text)` — 三层正则清理：
  - `data:image/png;base64,...`（data URI）→ 替换为 `[图片]`
  - `[A-Za-z0-9+/]{200,}={0,2}`（长 base64）→ 删除
  - `[0-9a-fA-F]{64,}`（长十六进制）→ 删除
- 在 `buildEmbedText()` 返回前调用，确保 BOW 向量只基于纯净文本

#### 7. LRU 检索缓存（性能优化）

**文件：** `internal/tools/knowledge.go`（+62 行）

**问题：** `searchMemoryFull` 每次都执行全库扫描 + BOW 计算，高频对话中产生多余磁盘 I/O。

**方案：**
- `searchCache` — `map[string]searchCacheEntry`，key = `query|limit|namespace`
- 容量 50 条，TTL 2 分钟，超限删最旧一条
- `sync.RWMutex` 读写分离，线程安全
- 跳过 trace 模式（调试数据不进缓存）

**决策：** 用 Go 原生 map + 手动 TTL 而非引入外部 LRU 库，因为 50 条级别不需要 O(1) 淘汰，手动清理最旧一条够用。

#### 8. Query 智能改写（检索质量）

**文件：** `plugins/think_plugin.go`（+73 行）、`plugins/search_plugin.go`（含在搜索相关性中）

**问题：** 用户口语化查询（如"昨天说的那个改动怎么样了"）直接检索会失焦，无法命中知识库中名为"20260521 修复 token 截断机制"的条目。

**方案（知识库检索）：**
- `needsQueryRewrite()` — 检测 20+ 个口语化模式（"昨天""上次""那个改动""怎么说的"等）
- `rewriteQuery(query, sessionID)` — LLM 改写 + 最近 3 轮对话上下文辅助，10s 超时自动降级
- 改写后的 query 用于检索，原 query 用于 LLM 回答（用户看到的还是对原问题的回答）

**方案（Web 搜索）：**
- `needsWebQueryRewrite()` — 检测口语化模式 + 查询过长（>30 字符）
- `rewriteWebQuery(query)` — LLM 精简为 5-15 字搜索关键词
- 改写后的关键词发给外部搜索 API

**决策：** 改写用 LLM 而非规则，因为口语化表达太多变，规则覆盖不了。10s 超时 + 静默降级确保不影响主流程。两处都走 `llm.RecordUsage` 记录 token 消耗。

#### 9. 结果低分过滤（检索质量）

**文件：** `internal/tools/knowledge.go`（+15 行）

**问题：** L0/L0.5/L2 三路召回后，低分噪音稀释上下文有效度。

**方案：**
- 排序后，丢弃 `score < 5` 的结果，但至少保留 1 条最高分
- 在排序和 limit 截断之间执行

#### 10. RSS 熔断器（鲁棒性）

**文件：** `internal/tools/rss.go`（+90 行）

**问题：** RSS 源（如掘金/InfoQ）经常反爬或下线，反复请求浪费时间和网络资源。

**方案：**
- `rssCircuitBreaker` — `map[string]*rssCircuitState`，跟踪每源连续失败次数
- 连续 5 次失败 → 标记 `offline`，24 小时内跳过该源
- 冷却结束后自动重试，成功则重置计数
- `sync.Mutex` 保护并发访问

### 八、Zero-Ops 自动合并逻辑

**问题**：knowledge_heal 每周扫描只生成建议，不自动执行。大量低风险重复条目需要人工处理。

**决策**：
- 添加 `auto_merge` 参数，仅对 HitCount=0 的条目自动合并
- HitCount=0 表示从未被检索命中，属于"沉睡"条目，合并风险极低
- 保留较长的 Summary 作为 target，较短的作为 source（被合并后删除）
- 被合并的 ID 记入 `mergedIDs` 集合，避免后续配对重复处理

**关键设计**：
```go
// 自动合并条件：两个条目 HitCount 均为 0
if autoMerge && a.entry.HitCount == 0 && b.entry.HitCount == 0 {
    sourceID, targetID := a.entry.ID, b.entry.ID
    if len(a.entry.Summary) > len(b.entry.Summary) {
        sourceID, targetID = b.entry.ID, a.entry.ID
    }
    KnowledgeMerge(sourceID, targetID)
}
```

**工作流更新**（`workflows/knowledge_heal.yaml`）：
```yaml
- id: scan
  inputs:
    threshold: 60
    auto_merge: true  # 新增：每周自动合并沉睡条目
```

### 十一、设计思考

**为什么 HitCount 是好的合并门禁？**
- HitCount=0 = 从未被用户检索到 = 系统认为"不重要"或"冗余"
- 合并这些条目不会降低检索质量（它们本来就没被检索到）
- 如果合并错误，用户下次检索时会发现缺失，可从 version history 恢复
- 这比纯相似度阈值更安全：相似度高但 HitCount 也高的条目可能是用户刻意分开的

**合并方向的选择**：
- 保留 Summary 更长的条目作为 target（信息更丰富）
- 但如果一个条目有更多 TypedLinks，应该保留它
- 当前简化为"保留更长的"，后续可加入 Links 数量加权

## 晚间第二次迭代：L3 vs L4 边界重新确立 + 洞察闭环

### 关键方向讨论

在做 `project_analyze` 工作流时，最初试图在 YAML 层解决 GitHub README 的获取问题——拼接 URL、on_error 链式降级。用户指出这应该是 L3 工具做的事，不是在 L4 工作流里打补丁。

**结论：遇到能力缺口时，优先考虑 L3（工具层）或 L4（插件层），而不是在 YAML 编排里做 URL 拼接、条件判断这类逻辑。**

这个讨论明确了整个项目的架构原则：

```
L1 内核（冻结）→ 路由转发
L2 glue IPC → 子进程管理（Python 插件）
L3 工具 → 数据接入 + 原子操作（65+ 工具）
L4 工作流 → 编排已有工具（32+ 工作流）
外部平台 → L3 工具接入 → L4 工作流编排 → 知识库
```

### github_readme L3 工具

**为什么做：** 工作流里用 `${input}/raw/main/README.md` 拼接 URL 是错误做法。URL 构造、分支降级（main→master）、错误处理应该是工具层的职责。

**做法：** `github.go`（新文件，~130 行）

- `extractGitHubRepo()` — 正则解析 GitHub 仓库 URL → owner/repo
- `fetchGitHubReadme()` — 构造 `raw.githubusercontent.com` URL，先试 main 分支，失败试 master
- 返回 `{title, content, size, repo}` 结构化 JSON
- 零 API key，纯 HTTP GET

### project_analyze 工作流重构

从 YAML 拼接 URL 改为 L3 工具调用：

```
修前：fetch_raw → on_error → fetch_page（URL 在 YAML 里拼接）
修后：github_readme（L3）→ on_error → web_fetch（L3）
```

### insight_followup 洞察闭环

**为什么做：** 分析出可借鉴的设计（如 OpenHuman 的 TokenJuice）后，只入库就结束了。没有把"可借鉴"变成"待办"。

**做法：** `insight_followup.yaml`（新工作流，6 步）

```
get_entry → evaluate（LLM 判断是否可落地）
  → create_todo（knowledge_add 创建改进建议条目）
  → add_todo（todo_plugin 添加待办）
  → link（knowledge_confirm_links 关联回源）
  → done（报告）
```

**实测：** OpenHuman 分析结果 → LLM 评估 actionable=true → 创建改进建议条目 → 关联回原始分析。

### 完整链路

```
agent_observer（每日 08:00 定时触发）
  → rss_default（5 源拉取）
  → LLM 筛选（多语言项目）
  → project_analyze（逐个项目，并行 2 个）
    → github_readme（L3，raw README）
    → LLM 深度分析（架构/技术栈/亮点/借鉴点）
    → knowledge_add（入库）
    → insight_followup（自动触发，6 步闭环）
      → LLM 评估 → 创建改进建议 → 添加待办 → 关联
```

### 核心认知

1. **L3 和 L4 的边界不是技术限制，是架构原则** — URL 拼接在 YAML 里"也能工作"，但把逻辑放在正确层级是项目长期健康的关键。L3 工具负责"怎么拿到数据"，L4 工作流负责"拿到数据后怎么做"。
2. **洞察不落地的分析是半截的** — `project_analyze` 最初只入库不行动。`insight_followup` 让分析结果变成可执行的待办，完成了"发现→理解→行动"的闭环。
3. **L3 工具是对外接口，不是内部实现** — `github_readme` 不只服务于 `project_analyze`，也可以被 LLM 直接调用（"帮我看看这个项目怎么样"），或者被其他工作流复用。
4. **知识库里不只有事实，还应该有"待办"** — source_type=note 的改进建议条目加上 todo_add，让知识库从"档案室"变成了"行动清单"。

## 第三方深度评估：DeepSeek 专家模式对比分析 beishan-core vs OpenHuman

| 维度 | beishan-core | OpenHuman |
|------|-------------|-----------|
| 核心目标 | **AI 的可靠**：可预测、零错误容忍、工程系统 | **AI 的有用**：主动了解用户、融入数字生活 |

分析指出，当大家都在为 AI 的"强大"欢呼时，beishan-core 在冷静地思考 AI 的"可靠"。硬化层不是限制，而是确保 AI 不失控的工程保障。这个方向在 AI 社区里不仅独特，而且弥足珍贵。

### TwinFlower 理论印证

beishan-core 和 OpenHuman 从架构哲学上完美印证了 TwinFlower 理论：

> **可靠的日常运行（左花）与持续的探索进化（右花），是 AI 系统不可分割的两面。**

- beishan-core 是方法论中的**左花（日常）**：稳定可靠、对错误零容忍
- OpenHuman 是方法论中的**右花（探索）**：广度连接、主动探索、个性进化

精妙之处在于实际角色发生了互换——beishan-core 是内部探索工具但偏左花架构，OpenHuman 是外部日常助手但偏右花架构。这证明 TwinFlower 理论描述的不是僵化标签，而是可灵活映射的动态结构关系。

硬化层解决"AI 如何可靠使用知识"，记忆树解决"AI 知道什么"。两者缺一不可。

### 第三次推演：外援分析 + 定位收敛

DeepSeek 专家模式给出完整对比分析：beishan-core 的硬化哲学是护城河，与 OpenHuman 的记忆树是硬币两面。同时印证了 TwinFlower 理论——beishan-core 是左花（日常/稳定），OpenHuman 是右花（探索/连接）。

带着 5 个问题请教外援，得到的结论：

```
之前错误的方案：
  beishan-core = 平台 + 编码 IDE（太重了）

正确的定位：
  beishan-core = 平台（硬化层 + 知识管理 + 工作流）
  编码是独立问题域，由外部工具负责
  beishan-core 只做编码安全工具
```

### 最终定位

| 不属于 beishan-core | 属于 beishan-core |
|--------------------|-------------------|
| 编码智能体 | code_security_check（扫描安全）|
| Claude CLI 集成 | code_apply（受控写入）|
| 自动改代码 | code_diff（对比变更）|
| 热更新 | code_read（读文件）|

### code_security_check L3 工具

第一个落地的编码安全工具，纯确定性逻辑：

**硬化边界确认：**

| 能做的 | 不做的 |
|--------|--------|
| ✅ 危险命令（rm -rf、os.RemoveAll 变量拼接） | ❌ SQL 注入（需语义理解）|
| ✅ 路径穿越（../） | ❌ 逻辑 bug（编译通过但行为不对）|
| ✅ 敏感路径写入（/etc/ /dev/ /proc/）| |
| ✅ 不安全权限（chmod 777）| |
| ✅ 自修改防护（写 kernel/ tools.go 等关键文件）| |
| ✅ 命令注入（exec.Command + 变量拼接）| |
| ✅ 硬编码密钥检测 | |

**实测效果：** `rm -rf` 和 `chmod 777` 同时被 `block` 级别拦截。

**架构原则（第三次推演的最终结论）：**

1. **beishan-core 的核心壁垒是硬化层 + 知识管理 + 工作流编排**，不是编码能力
2. **L3 工具做数据接入 + 原子操作，L4 工作流做编排**，不在 YAML 里打补丁
3. **外部工具（Claude CLI / Cursor / Codex）只做代码生成**，不接触知识库
4. **知识体系独立** — beishan-core 的知识库不接收外部 AI 的知识文件

### 编码安全工具集落地

定位收敛后，4 个 L3 工具按顺序实现：

| 工具 | 硬化边界 | 行数 |
|------|---------|:----:|
| `code_security_check` | 危险命令、路径穿越、敏感路径、自修改防护、命令注入、硬编码密钥 | 227 |
| `code_read` | 路径规范化、穿越检测、范围限制、1MB 上限、纯内容返回 | 207 |
| `code_apply` | 安全前置检查、自动备份（.backup/）、自修改防护、回滚支持 | 185 |
| `code_diff` | 复用硬化路径校验，输出 added/removed/unchanged 统计 |（同 code_read）|

**code_read 硬化链：**

```
用户输入路径
  → 展开 ~/ → filepath.Clean 规范化
  → 检测 .. 路径穿越
  → 拼接项目根目录（相对路径转绝对）
  → filepath.Rel 检测是否在根目录下
  → os.Stat 检查存在性 + 非目录
  → 1MB 大小限制
  → os.ReadFile 读取
  → 返回纯内容
```

**code_apply 硬化链：**

```
用户输入路径 + 内容
  → 路径规范化（同 code_read）
  → 自修改防护：匹配 kernel/ tools.go 等关键文件则拒绝
  → code_security_check 扫描内容
  → 自动备份原文件到 .backup/（含时间戳）
  → os.WriteFile 写入
  → code_rollback 可回滚到最新备份
```

**实测：**
- `code_read internal/tools/code_read.go` → 207 行，5088 bytes ✅
- `code_diff` 对比 code_security.go 和 code_read.go → +199/-215 行 ✅
- `code_apply test_write.txt "hello world"` → 写入 11 bytes ✅
- `code_apply /tmp/test.txt` → 路径超出项目范围，已拦截 ✅
